# 评估模块：实验设计与评价指标

## 1. 实验设计目标

MalBlock 并非传统的二分类检测模型，而是一个面向恶意流量处置闭环的研究系统。系统从流量输入出发，经过告警抽取、时间窗聚合或结构化流适配、RAG 辅助检索、LLM 策略生成以及 nftables 执行，最终形成“检测—决策—执行—评估”的完整链路。因此，系统评估不能仅停留在“识别准确率”层面，还需要同时考察策略合理性、执行落地情况、安全约束满足情况以及阻断后的收益表现。

基于这一研究目标，本文在评估模块中构建了多维度指标体系，对系统输出进行统一评价。相关评估逻辑统一由 `backend/eval/channel_eval.py:1051` 中的 `evaluate_job()` 完成，评估入口脚本为 `backend/scripts/evaluate_channel.py:16`。评估结果统一保存为 `evaluation_report.json`，以支持离线复盘、前端展示和论文分析。

---

## 2. 实验对象与通道设计

本文系统支持三类实验通道：

1. **CSV 结构化流级通道**  
   直接使用结构化流记录作为输入，适合在理想特征输入条件下评估 LLM 的风险识别和策略选择能力。由于该通道保留了流级标签信息，因此可进行基于真值标签的量化分析。

2. **离线 PCAP 通道**  
   通过 Suricata 对离线 PCAP 进行检测，并将告警聚合为时间窗后送入 LLM 决策。该通道主要用于复盘完整闭环流程，评估策略合理性、执行表现以及代理性收益。

3. **在线 replay 通道**  
   将 PCAP 在实验网络环境中回放，由 Suricata 实时监听并持续产生告警，系统按滚动窗口触发决策与执行。该通道主要用于验证联机场景下的处置效果、执行稳定性和阻断收益观察。

由于三类通道的数据形态和标签条件不同，本文采用“统一框架、分通道适配”的评估方式：对于 CSV 通道，重点评估决策正确性；对于 offline/replay 通道，重点评估策略智能性、执行有效性、安全可控性和收益代理指标。

## 3. 评价指标体系

结合系统“可运行、可解释、可执行、可评估”的研究目标，本文将评估划分为以下六个方面。

### 3.1 数据覆盖性评估

数据覆盖性评估用于描述当前实验任务所覆盖的数据规模和样本分布，其实现位于 `backend/eval/channel_eval.py:311`。该部分主要回答“本次实验到底评了哪些数据、覆盖是否充分”的问题。

对于 CSV 通道，数据覆盖性指标包括：

- 输入总样本数（`input_rows`）
- 参与评估的采样样本数（`selected_rows`）
- 恶意样本数与良性样本数
- 标签分布（`label_distribution`）
- 攻击族分布（`attack_family_distribution`）
- 来源日期分布（`source_day_distribution`）
- 唯一源 IP 数量
- 唯一流标识数（`unique_flow_uid_count`）
- 重复输入数（`duplicate_input_count`）

对于 offline/replay 通道，由于评估对象是告警聚合后的窗口而非原始带标签流记录，因此主要统计 surviving windows 和 selected windows，以反映告警窗口筛选规模及样本覆盖情况。

该部分指标的意义在于避免实验结果只建立在少数样本或少数攻击类型之上，从而增强实验结论的代表性与说服力。

### 3.2 决策有效性评估

决策有效性评估主要面向 CSV 通道，其核心实现位于 `backend/eval/channel_eval.py:835`。由于 CSV 输入保留了流级标签信息，因此可以将 LLM 决策结果与原始样本真值进行对齐分析。对齐逻辑由 `_csv_eval_key_from_input()` 与 `_csv_eval_key_from_decision()` 完成，见 `backend/eval/channel_eval.py:252` 和 `backend/eval/channel_eval.py:281`。

考虑到本系统并非简单二分类器，而是输出多级响应动作，因此本文设计了两套决策效果指标：

#### （1）风险识别指标（Risk Detection Metrics）

该指标将 `block`、`monitor` 和 `review` 均视为“识别到风险”，只有 `allow` 视为未识别风险。其目的在于衡量系统是否能够对恶意流量做出风险感知，而非误将其直接放行。

对应统计包括：

- TP：恶意样本被识别为风险
- FP：良性样本被误识别为风险
- TN：良性样本被正确放行
- FN：恶意样本被错误放行

并进一步计算：

- Precision
- Recall
- F1
- Accuracy

#### （2）强处置指标（Strong Mitigation Metrics）

该指标仅将 `block` 视为正类，用于衡量系统对恶意流量实施强阻断的能力。其本质是从“风险识别”进一步提高到“强干预执行”的评价层面。

与风险识别指标相比，强处置指标更适合衡量系统是否真正具备阻断能力，而不是仅能给出谨慎性的观察或复核建议。

此外，CSV 通道还会输出如下细粒度分析结果：

- 按标签统计的指标（`per_label_metrics`）
- 按攻击族统计的指标（`per_attack_family_metrics`）
- 按来源日期统计的指标（`per_source_day_metrics`）

这使得本文能够分析模型在不同攻击类型和不同数据来源条件下的稳定性与泛化情况。

### 3.3 策略智能性评估

策略智能性评估是本文区别于传统检测实验的关键部分，其实现位于 `backend/eval/channel_eval.py:628`。该部分并不只关心“判得对不对”，而是进一步评价“给出的策略是否合理”。

为此，本文在评估模块中构建了基于专家知识的规则库（`EXPERT_RULES`），见 `backend/eval/channel_eval.py:73`。规则库按照攻击族划分，包括 DoS、Botnet、Brute Force、Web Attack、Scan、Suspicious DNS、TCP Anomaly 等多种类型，并进一步按照风险等级划分为高、中、低三级。对于每一类规则，系统预定义：

- 允许的动作集合
- 允许的执行模式
- 合理的处置强度范围
- 推荐动作与推荐执行模式
- 规则依据说明

在此基础上，评估模块首先根据证据字段、噪声比例、严重度、告警次数和历史递进标签等因素推断当前决策所属的风险等级，见 `backend/eval/channel_eval.py:567`；再将实际决策与专家规则进行比对，输出以下指标：

- **策略匹配率**（`strategy_match_rate`）：动作与执行模式同时符合专家规则的比例
- **动作匹配率**（`action_match_rate`）：动作是否合理
- **执行模式匹配率**（`execution_mode_match_rate`）：执行方式是否合理
- **过度处置率**（`over_mitigation_rate`）：处置强度超过合理上限的比例
- **处置不足率**（`under_mitigation_rate`）：处置强度低于合理下限的比例
- **TTL 自适应分数**（`ttl_adaptation_score`）：高风险决策的封禁时长是否整体大于中低风险决策
- **升级一致性**（`escalation_consistency`）：同一源 IP 重复出现风险时，处置强度是否保持一致或递进升级

这一部分指标能够体现 LLM 策略输出是否具备“像安全专家一样决策”的能力，也能体现系统在重复攻击和多级响应场景下的策略一致性。

### 3.4 执行有效性评估

执行有效性评估用于衡量 LLM 输出的策略是否真正被执行层正确落地，其实现位于 `backend/eval/channel_eval.py:387`。

本文系统并不将模型输出视为最终结果，而是要求其进一步映射到执行工具，例如：

- `block + drop` 对应 `block_ip`
- `block + rate_limit` 对应 `rate_limit_ip`
- `monitor + watch` 对应 `watch_ip`

因此，执行层评估主要包括以下指标：

- 工具调用成功次数（`tool_success_count`）
- 工具调用失败次数（`tool_failure_count`）
- 新增执行次数（`new_enforcement_count`）
- 重复执行或被已有策略覆盖的次数（`repeat_enforcement_count`）
- 已存在记录数（`already_present_count`）
- 短路跳过次数（`skipped_execution_count`）
- 被已有更强动作覆盖次数（`covered_by_existing_action_count`）
- 决策—执行一致性（`decision_to_execution_consistency`）
- 唯一被阻断/限速/观察的 IP 数量

其中，“决策—执行一致性”是一个关键指标，用于衡量模型的策略语义是否与工具执行结果保持一致。如果模型决定了某种动作，但执行层并未产生相应工具行为，则说明闭环存在偏差。

该部分评估能够证明本文系统并非停留在“文本化建议”层面，而是真正完成了从决策到执行的闭环落地。

### 3.5 安全可控性评估

由于本文将大模型引入网络安全处置链路，因此系统的安全边界和约束满足情况必须单独评估。相关实现位于 `backend/eval/channel_eval.py:962`，所依据的基础安全约束定义在 `backend/agent/policy.py:114`。

安全可控性评估主要包括：

- **受保护 IP 误阻断尝试次数**（`protected_ip_block_attempt_count`）
- **受保护 IP 的安全处理次数**（`protected_ip_safe_handling_count`）
- **纯噪声误阻断次数**（`noise_only_false_block_count`）
- **高噪声过激处置次数**（`high_noise_overreaction_count`）
- **Fallback 触发次数**（`fallback_trigger_count`）
- **约束违规数**（`constraint_violation_count`）
- **预检干预次数**（`precheck_intervention_count`）

通过这些指标，本文能够说明：即使系统引入了 LLM 参与安全决策，其实际执行仍然受到白名单、约束校验、运行时预检和保守回退机制的限制，从而保证系统总体可控。

### 3.6 干预收益观察评估

在 offline 和 replay 通道中，本文进一步设计了“干预收益观察”指标，用于衡量执行动作之后后续风险窗口是否出现下降趋势。其实现位于 `backend/eval/channel_eval.py:779`。

需要强调的是，该指标并非严格因果推断，而是一个**代理指标**（proxy metric），代码中显式标注为 `metric_semantics = "proxy"`。其基本思想为：

1. 对每个采用 `drop` 或 `rate_limit` 的决策，找到其对应的当前风险窗口；
2. 在该决策生效 TTL 期间，统计同一源 IP 后续窗口中的风险告警量；
3. 将后续风险强度与决策前窗口风险强度进行比较，计算风险压制比例。

计算结果包括：

- 参与评估的决策数
- 平均风险压制比例（`mean_suppression_ratio`）
- 每条决策对应的前后对比样本

由于 offline/replay 场景缺乏严格的真实干预对照组，因此该指标只能作为“处置后风险下降趋势”的代理性观察结果，而不应被表述为绝对的因果收益结论。

对于 CSV 通道，由于其本质是结构化静态流记录评估，不存在自然的时间连续性，因此该收益指标被标记为“不适用”。

## 4. 误差分析与样本复盘

除了定量指标外，本文还在评估报告中保留了错误样本与关键案例，用于定性分析和论文展示，包括：

- 假阳性样本
- 假阴性样本
- review 样本
- 未匹配样本
- 策略不匹配样本
- 约束违规样本
- 工具失败样本

这些样本由 `error_analysis` 字段统一保存，可为后续论文中的案例分析、系统局限性讨论和前端可视化提供依据。

## 5. 本文评估体系的特点

综上，本文所设计的评估体系不是单一的分类性能测试，而是围绕 LLM 恶意流量阻断系统的完整闭环构建，具体体现为：

1. 从**数据覆盖性**角度说明实验对象是否具有代表性；
2. 从**决策有效性**角度衡量系统对风险的识别能力与强处置能力；
3. 从**策略智能性**角度评价大模型输出是否符合安全专家知识；
4. 从**执行有效性**角度验证策略是否真正落地执行；
5. 从**安全可控性**角度确保系统不会因引入大模型而失控；
6. 从**干预收益观察**角度评估执行后风险窗口是否呈下降趋势。

因此，本文评估体系能够较为完整地反映 MalBlock 系统在“识别—决策—执行—反馈”全过程中的综合表现。

## 6. 评估模块到底做了哪几个方面的评估

为了便于答辩和汇报，可以将评估模块概括为以下 **六个核心方面**：

1. **数据覆盖性评估**  
   回答“这次实验到底评了多少数据、覆盖了哪些标签和攻击类型”。

2. **决策有效性评估**  
   回答“LLM 有没有识别出风险、有没有真正给出强阻断动作”。

3. **策略智能性评估**  
   回答“模型生成的动作、执行模式和 TTL 是否符合安全专家规则”。

4. **执行有效性评估**  
   回答“模型决定之后，执行层有没有真正落地，且是否与决策保持一致”。

5. **安全可控性评估**  
   回答“系统会不会误封关键资产、会不会对噪声流量过激响应，整体是否可控”。

6. **干预收益观察评估**  
   回答“执行阻断之后，后续风险窗口强度是否出现下降趋势”。

此外，评估模块还辅以**误差分析与样本复盘**，用于定位假阳性、假阴性、策略失配与工具失败等典型问题。
