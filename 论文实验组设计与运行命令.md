# MalBlock 论文实验组设计与运行命令

本文档基于当前项目代码实现，按照你当前论文定位重新整理实验方案。

**核心定位：**

> 本系统并不把大模型作为恶意流量检测器使用，而是将 **Suricata 等安全检测工具产生的告警** 作为输入证据，再由 **大模型负责阻断决策、执行方式选择与响应控制**。

因此，论文实验的核心不应围绕“LLM 能否识别恶意流量”展开，而应围绕以下问题展开：

1. **大模型是否能做出合理的阻断决策；**
2. **RAG 是否能提升阻断决策质量；**
3. **系统是否能完成检测—决策—执行闭环；**
4. **执行阻断后，后续风险是否得到压制；**
5. **在线回放场景下，系统是否能稳定联动运行。**

---

## 1. 论文中建议保留的正式实验组

基于当前系统定位，建议论文主实验聚焦为 **4 组核心实验 + 1 组可选补充实验**。

| 实验组 | 名称 | 通道 | 作用 | 是否建议写入主结果 |
|---|---|---|---|---|
| G1 | RAG 消融基线实验（无 RAG） | CSV | 评估纯阻断决策基线能力 | 是 |
| G2 | RAG 增强实验（有 RAG） | CSV | 验证历史案例检索是否提升阻断决策质量 | 是 |
| G3 | Offline 阻断决策闭环实验 | Offline PCAP | 验证在真实检测输入下的阻断决策与执行闭环 | 是 |
| G4 | Replay 在线联动实验 | Replay | 验证系统在线回放场景下的联动运行能力 | 是 |
| G5 | 执行 / 无执行 对照实验 | Replay | 验证真实执行阻断后是否压制后续风险 | 是 |
| S1 | Top-k 工程补充实验（可选） | Offline / Replay | 只用于说明 token 成本优化 | 可选附录 |

---

## 2. 为什么按这个思路设计实验

### 2.1 你的论文重点是“阻断决策”，不是“恶意流量检测”

在当前系统中：

- **Suricata** 负责发现告警；
- **聚合模块** 负责把告警整理成可供决策的时间窗证据；
- **LLM Agent** 负责根据已有证据决定：
  - 是 `block` 还是 `monitor` / `review`
  - 是 `drop` 还是 `rate_limit` / `watch`
  - 阻断 TTL 应该设置多久
  - 是否存在过度处置或处置不足

因此论文最应该证明的是：

> **大模型能否对已发现的恶意流量做出更合理的处置决策，并在系统中完成有效执行。**

而不是证明：

> 大模型能否替代 Suricata 识别恶意流量。

---

### 2.2 为什么不再把 top-k 消融当主实验

`topk` 在你系统里本质上是一个**工程优化参数**，其最初目标是：

- 减少传入 LLM 的消息数量
- 降低 token 开销
- 控制推理成本
- 避免在线通道调用量过大

这不是你论文的主要科学问题，因此：

- `topk` **不必作为主实验核心内容**
- 如果需要，可以只作为附录中的工程优化补充
- 主实验应更多集中于：
  - RAG 是否提升决策
  - 阻断是否合理
  - 执行是否有效
  - 执行后风险是否下降

## 3. 每组实验到底回答什么问题

## G1：RAG 消融基线实验（无 RAG）

### 目标
回答：

> 在没有历史案例检索时，大模型仅依据当前告警证据，能否做出合理的阻断决策？

### 作用
这是整篇论文的**阻断决策基线实验**。没有这组，就无法证明 RAG 是否有效。

### 推荐设置
- 通道：`csv`
- 数据：`mixed_eval_cleaned.csv`
- `topk=0`，确保所有选中样本进入 LLM
- `selection_mode=priority`
- `rag_top_k=0`，关闭 RAG

### 核心指标
建议重点看：

- `strategy_eval.strategy_match_rate`
- `strategy_eval.action_match_rate`
- `strategy_eval.execution_mode_match_rate`
- `strategy_eval.over_mitigation_rate`
- `strategy_eval.under_mitigation_rate`
- `safety_eval.constraint_violation_count`
- `safety_eval.high_noise_overreaction_count`
- `execution_eval.repeat_enforcement_ratio`

### 说明
这一组实验不应被表述为“恶意流量识别基线”，而应写成：

> **无 RAG 条件下的大模型阻断决策基线性能。**

---

## G2：RAG 增强实验（有 RAG）

### 目标
回答：

> 引入历史案例检索后，大模型是否能做出更稳定、更保守或更符合经验规则的阻断决策？

### 作用
这是论文中最重要的**方法有效性实验**之一，用于验证 RAG 是否真正帮助了“阻断决策”。

### 推荐设置
- 与 G1 保持完全一致
- 唯一变化：`rag_top_k=3`

### 核心对比指标
与 G1 对比：

- `strategy_match_rate`
- `action_match_rate`
- `execution_mode_match_rate`
- `over_mitigation_rate`
- `under_mitigation_rate`
- `constraint_violation_count`
- `high_noise_overreaction_count`

### 论文建议表述
这一组用于说明：

> **RAG 有助于提升 LLM 在阻断决策场景下的经验一致性、约束遵守能力与处置稳定性。**

---

## G3：Offline 阻断决策闭环实验

### 目标
回答：

> 在真实 PCAP 经 Suricata 检测后，大模型是否能够基于检测证据完成阻断决策，并在执行层形成完整闭环？

### 为什么必须跑
CSV 实验主要说明“结构化证据输入下的决策质量”；而 Offline PCAP 实验才能说明：

- 真实检测输入是否可用
- 聚合后的证据是否足以驱动决策
- 决策是否能够落实到执行层
- 系统是否具有完整闭环能力

### 推荐设置
- 通道：`offline`
- `topk=0`，保证阻断决策评估更完整
- `rag_top_k=3`
- 建议至少跑以下 2~3 个代表性 PCAP：
  - `Tuesday-WorkingHours.pcap`
  - `Wednesday-workingHours.pcap`
  - `Thursday-WorkingHours.pcap`

### 核心指标
建议重点看：

- `decision_eval.action_distribution`
- `strategy_eval.strategy_match_rate`
- `strategy_eval.execution_mode_match_rate`
- `execution_eval.tool_success_count`
- `execution_eval.repeat_enforcement_ratio`
- `execution_eval.decision_to_execution_consistency`
- `safety_eval.precheck_intervention_count`
- `safety_eval.constraint_violation_count`
- `effect_eval.mean_suppression_ratio`

### 文件依据
除了 `evaluation_report.json` 外，还要结合：

- `channel_summary.json`
- `summary.json`

因为这些文件中保存了：

- 原始/过滤后告警数
- 聚合窗口数
- selected window 数
- 流式或离线运行摘要

### 论文建议表述
这一组应写成：

> **基于真实检测输入的阻断决策闭环实验**

而不是“恶意流量识别实验”。

---

## G4：Replay 在线联动实验

### 目标
回答：

> 在在线回放场景下，系统能否完成“回放—检测—决策—执行”的实时联动？

### 作用
这是论文中用来证明系统**在线可运行性**与**联动能力**的关键实验。

### 推荐设置
- 通道：`replay`
- 回放速率：`10x`
- `topk=20`
- `rag_top_k=3`
- 建议使用相对稳定、便于演示的 PCAP，例如：
  - `Thursday-WorkingHours.pcap`

### 为什么这里保留 `topk=20`
因为在线回放实验的核心目标不是“最完整地评估所有窗口”，而是：

- 验证联机条件下系统是否能工作
- 控制 LLM 调用量
- 避免回放链路因为消息过多而拖慢
- 体现工程可部署性

因此这里保留 `topk=20` 是合理的，不影响论文主线。

### 核心指标
建议看：

- `stream_alert_count_raw`
- `stream_alert_count_filtered`
- `stream_batches`
- `selected_groups`
- `decision_count`
- `execution_eval.tool_success_count`
- `execution_eval.repeat_enforcement_ratio`
- `nft` 实际落表情况

### 论文建议表述
这一组应写成：

> **在线回放环境下的阻断决策联动验证实验**

---

## G5：执行 / 无执行 对照实验

### 目标
回答：

> 在同样的回放流量、同样的参数条件下，真实执行阻断后，后续风险告警是否较“不执行”场景明显下降？

### 为什么这组实验非常重要
这是支撑“阻断决策真的产生收益”的关键证据。

如果没有这组实验，你只能说：

- 系统做了阻断
- nftables 落表了

但还不能更有说服力地说明：

- **阻断后后续风险确实被压制了**

### 实验设计方式
建议做两次 Replay 实验：

#### 执行组
- 真实执行 nft
- `unset DRY_RUN`

#### 对照组
- 不真实执行 nft
- `export DRY_RUN=1`

两组必须保持：

- 同一个 PCAP
- 同一个 replay 速率
- 同一个 `window_sec`
- 同一个 `min_hits`
- 同一个 `topk`
- 同一个 `rag_top_k`
- 同一个 replay 环境

唯一差异：

- 是否执行 nft 阻断

### 核心对比指标
建议基于：

- 执行组的 `llm_decisions.jsonl`
- 两组各自的 `alerts_filtered.jsonl`

做对齐比较，重点统计：

1. 同一 `src_ip` 在决策后 TTL 范围内的后续告警数
2. 执行组 vs 对照组的后续告警 hits 差异
3. 平均压制增益
4. 完全压制率（后续告警为 0 的比例）

### 论文建议表述
这一组用于说明：

> **阻断决策不仅能够被执行，而且执行后能有效降低后续被 Suricata 观测到的风险告警强度。**

## S1：Top-k 工程补充实验（可选）

### 是否必须
**不是必须。**

### 适用场景
只有在你希望补充说明“为什么在线通道不把所有窗口都送入 LLM”时，才建议把它作为附录中的小实验。

### 推荐对比方式
同一个 PCAP，只比较：

- `topk=0`
- `topk=20`

只说明：

- 进入 LLM 的窗口数变化
- 运行时长变化
- token / 调用成本变化（如果你能统计）

这一组不建议作为主结果表核心内容。

```bash
export DRY_RUN=1
```

### RAG 运行前检查

你之前已经出现过 RAG embedding 超长报错，因此正式跑 G2 / G3 / G4 / G5 之前，建议确认：

1. 已修复 `backend/agent/rag_store.py` 的超长文本问题；或者
2. 本次运行中未再次出现：
   - `[rag] vector retrieval failed`
   - `[rag] vector upsert failed`

如果仍然出现这些报错，则论文中不应直接写成“向量 RAG 正常生效”，而应谨慎说明存在 fallback 情况。

建议顺序如下：

## 10. 一句话结论（适合写进论文实验设计章节）

本文实验体系围绕“**大模型辅助恶意流量阻断决策**”展开，而非围绕恶意流量检测展开：其中，RAG 消融实验用于验证历史案例检索对阻断决策质量的提升作用，Offline 阻断决策闭环实验用于验证真实检测输入下的系统完整闭环能力，Replay 在线联动实验用于验证联机场景下的系统运行能力，执行 / 无执行对照实验用于验证真实执行阻断后后续风险告警是否得到压制。
