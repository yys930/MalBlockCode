# 论文正式实验设计方案（三模型版）

本文档用于指导毕业论文正式实验。设计目标是：

1. 保证实验符合本科毕业设计要求；
2. 保证实验具有基本科研规范与可复现性；
3. 保证论文主线清晰，实验规模适中但结论完整；
4. 让你在完成实验后，可以直接进入论文写作阶段。

---

## 1. 论文实验总思路

本论文的核心不是证明“大模型能否检测恶意流量”，而是证明：

> 在 Suricata 等检测工具已经给出告警证据后，大模型能否做出有效的阻断决策，并驱动系统完成真实执行与风险压制。

因此，实验设计分成两部分：

### 第一部分：三模型横向对比
用于比较 **模型1 / 模型2 / 模型3** 的阻断决策能力。

### 第二部分：最优模型系统验证
从三模型中选出综合表现最好的一个模型，再完成系统级实验，验证闭环执行、在线联动和干预收益。

这种设计最适合本科毕设：
- 既有模型对比；
- 又有系统验证；
- 实验规模合理；
- 论文结构清楚，不会显得杂乱。

---

## 2. 论文采用的评估维度

当前论文正式保留以下评估维度：

1. **决策有效性评估**
2. **执行有效性评估**
3. **干预收益观察评估**
4. **数据覆盖性评估（可选支撑项）**

对应关系如下：

- **CSV 通道**：主要做决策有效性评估
- **Offline 通道**：主要做执行有效性评估、干预收益观察评估
- **Replay 通道**：主要做执行有效性评估、干预收益观察评估、联动能力验证
- **数据覆盖性评估**：三种通道都可保留

## 4. E1：三模型无 RAG 决策有效性对比

比较模型1、模型2、模型3在 **没有 RAG** 的条件下，仅依据当前告警证据做阻断决策时的效果。

## 4.3 数据
使用：

```text
/home/os/FinalCode/malblock/backend/datasets/cic_ids2017_trafficlabelling/mixed_eval_cleaned.csv
```

## 4.4 推荐参数
- `topk=0`（全量运行，注意是显式写 `--topk 0`，不是省略 `--topk`）

## 4.6 重点评估指标

- `decision_eval.risk_detection_metrics.f1`
- `decision_eval.strong_mitigation_metrics.precision`
- `decision_eval.strong_mitigation_metrics.recall`
- `decision_eval.strong_mitigation_metrics.f1`

- `dataset_summary.selected_rows`

## 4.7 论文作用

## 5. E2：三模型有 RAG 决策有效性对比

## 5.1 实验目的
比较模型1、模型2、模型3在 **加入 RAG** 后的阻断决策效果，并观察 RAG 是否稳定提升模型表现。

## 5.2 通道
- **CSV 通道**

## 5.3 数据
与 E1 保持完全一致：

```text
/home/os/FinalCode/malblock/backend/datasets/cic_ids2017_trafficlabelling/mixed_eval_cleaned.csv
```

## 5.4 推荐参数
除 `rag_top_k` 外，与 E1 必须完全一致：
- `topk=0`（全量运行，注意是显式写 `--topk 0`，不是省略 `--topk`）
- `selection_mode=stratified_label`
- `seed=42`
- `rag_top_k=3`
- `TEMP=0.0`（若代理不支持，可统一用 `0.1`）

## 5.5 实验次数
共 3 次：
- 模型1：1 次
- 模型2：1 次
- 模型3：1 次

## 5.6 重点评估指标
与 E1 相同，重点比较：
- `risk_detection_metrics.f1`
- `strong_mitigation_metrics.f1`
- `matched_decisions`

## 5.7 关键规范要求
正式实验时，三模型必须使用 **同一份固定的 RAG 知识库快照**。

也就是说，不能出现：
- 模型1 先跑后写入新案例；
- 模型2 再跑时读到了模型1新增案例；
- 模型3 又在更大的历史库上运行。

当前代码会在运行过程中自动向 RAG 写回新案例，因此正式实验不能让三个模型共用同一个活动中的 RAG 库连续运行。

### 正确做法：先冻结母版快照，再给每个模型恢复独立副本

#### 第一步：冻结正式实验母版快照（只做一次）

```bash
cd /home/os/FinalCode/malblock
source backend/MBvenv/bin/activate
mkdir -p backend/rag/formal_snapshot
rsync -a --delete backend/rag/chroma_db/ backend/rag/formal_snapshot/chroma_db/
cp backend/rag/decision_history.jsonl backend/rag/formal_snapshot/decision_history.jsonl
```

完成后，以下两项就是正式实验固定母版：
- `backend/rag/formal_snapshot/chroma_db`
- `backend/rag/formal_snapshot/decision_history.jsonl`

#### 第二步：模型1运行前恢复独立副本

```bash
rm -rf backend/rag/formal_runs/e2_model1
mkdir -p backend/rag/formal_runs/e2_model1
rsync -a --delete backend/rag/formal_snapshot/chroma_db/ backend/rag/formal_runs/e2_model1/chroma_db/
cp backend/rag/formal_snapshot/decision_history.jsonl backend/rag/formal_runs/e2_model1/decision_history.jsonl

export RAG_DB_DIR=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model1/chroma_db
export RAG_ARCHIVE_PATH=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model1/decision_history.jsonl
```

#### 第三步：模型2运行前恢复独立副本

```bash
rm -rf backend/rag/formal_runs/e2_model2
mkdir -p backend/rag/formal_runs/e2_model2
rsync -a --delete backend/rag/formal_snapshot/chroma_db/ backend/rag/formal_runs/e2_model2/chroma_db/
cp backend/rag/formal_snapshot/decision_history.jsonl backend/rag/formal_runs/e2_model2/decision_history.jsonl

export RAG_DB_DIR=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model2/chroma_db
export RAG_ARCHIVE_PATH=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model2/decision_history.jsonl
```

#### 第四步：模型3运行前恢复独立副本

```bash
rm -rf backend/rag/formal_runs/e2_model3
mkdir -p backend/rag/formal_runs/e2_model3
rsync -a --delete backend/rag/formal_snapshot/chroma_db/ backend/rag/formal_runs/e2_model3/chroma_db/
cp backend/rag/formal_snapshot/decision_history.jsonl backend/rag/formal_runs/e2_model3/decision_history.jsonl

export RAG_DB_DIR=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model3/chroma_db
export RAG_ARCHIVE_PATH=/home/os/FinalCode/malblock/backend/rag/formal_runs/e2_model3/decision_history.jsonl
```

### 为什么这样做
这样可以保证：
1. 三个模型都从同一份固定 RAG 母版开始；
2. 每个模型运行时写回的新案例只会污染自己的副本；
3. 三模型之间不会互相泄漏信息；
4. 对比结果更公平，也更符合科研规范。

### 论文中可直接这样写

> 为保证 RAG 对比实验公平性，正式实验前对历史案例库进行了快照冻结；模型1、模型2、模型3分别从同一份固定快照恢复独立工作副本后再运行，避免运行过程中新增案例对后续模型造成信息泄漏。

## 5.8 论文作用
这一组用于回答：

> 加入 RAG 后，不同模型的阻断决策效果是否提升，提升幅度是否一致。

## 6. 最优模型选择规则

完成 E1 和 E2 后，从模型1、模型2、模型3中选出 **综合最优模型**，用于后续系统级实验 E3、E4、E5。

建议采用以下优先级选择：

1. **strong_mitigation_metrics.f1**
2. **risk_detection_metrics.f1**
3. **matched_decisions**
4. 如前述指标非常接近，则选择输出更稳定、明显异常更少的模型

建议不要单纯以某一个 Precision 或 Recall 最高就直接定最优模型，而要综合看整体表现。

## 7.2 通道
- **Offline PCAP 通道**

## 7.3 数据

1. `Tuesday-WorkingHours.pcap`
2. `Wednesday-workingHours.pcap`
3. `Thursday-WorkingHours.pcap`


验证最优模型在在线回放场景下，能否稳定完成“回放—检测—决策—执行”的联动流程。

- `replay-interface=veth-mb-replay`

## 10. 为什么不建议三模型都跑所有系统实验

建议你：
- 三模型只做 **E1、E2** 的横向比较；
- 系统级实验 **E3、E4、E5** 只用最优模型完成。

原因如下：

1. **CSV 通道有真值标签**，最适合做模型间公平比较；
2. Offline / Replay 更适合做系统验证，而不是大规模模型横向比较；
3. 如果三模型全部展开所有实验，实验矩阵会过大，论文主线会被冲散；
4. 对本科毕设来说，这样做既全面，又不会超出合理工作量。

## 11. 实验公平性与科研规范要求

正式实验时，以下条件必须尽量固定：

1. **除模型名外，其他参数全部固定**；
2. CSV 实验必须使用相同数据、相同采样方式、相同随机种子；如采用全量评估，需显式写 `--topk 0`；
3. RAG 对比实验必须使用同一份固定知识库快照；
4. Replay 对照实验必须使用相同 PCAP、相同回放速率、相同窗口参数；
5. 正式模型对比建议统一 `TEMP=0.0` 或统一 `TEMP=0.1`；
6. 每次正式实验都要保留原始结果文件，不能只抄最后指标；
7. 如果某次实验失败，不要直接丢弃，需记录失败原因并重跑。

这些要求能保证实验结果具备基本的可信度和可复现性。

---

## 12. 建议的正式实验顺序

建议按以下顺序推进：

1. **先跑 E1**：比较三模型无 RAG 基础决策能力；
2. **再跑 E2**：比较三模型有 RAG 决策效果；
3. **选出综合最优模型**；
4. **跑 E3**：验证 Offline 闭环；
5. **跑 E4**：验证 Replay 在线联动；
6. **跑 E5**：完成执行 / 无执行对照实验。

## 15. 一句话总结
