# MalBlock 系统架构说明

## 1. 系统整体设计思路

### 1.1 系统目标
MalBlock 是一个面向恶意流量阻断场景的研究型系统。它的核心目标不是只做“恶意/正常”的识别，而是形成一条从输入流量到执行阻断策略的完整闭环。

系统的总体目标可以概括为三点：

1. 将原始流量或结构化流量记录转换为可供决策的安全证据。
2. 利用 RAG + LLM 生成结构化阻断策略，而不是只输出分类标签。
3. 通过真实的 `nftables` 执行阻断、限速或观察动作，并对整个过程做结构化评估和可视化展示。

从设计理念上看，这个系统不是单点模型验证，而是一个多层协作的实验平台。

### 1.2 整体架构分层
系统可以划分为以下几个层次：

1. **输入层**
   - 离线 PCAP 输入
   - 在线回放流量输入
   - CSV 结构化流量输入

2. **检测与预处理层**
   - 使用 Suricata 对 PCAP 或回放流量进行检测
   - 从 `eve.json` 中抽取告警
   - 做噪声过滤、字段清洗和时间窗聚合

3. **决策层**
   - 将聚合后的窗口证据转为统一 message
   - 注入系统约束、历史上下文和 RAG 检索结果
   - 用 LLM 生成结构化处置策略

4. **执行层**
   - 使用 MCP 调用执行器
   - 将策略写入 `nftables`
   - 执行 `drop`、`rate_limit` 或 `watch`

5. **评估与展示层**
   - 统计决策与执行结果
   - 生成结构化评估报告
   - 通过前端控制台展示运行状态、实验结果和 nftables 实时状态

### 1.3 系统总体流程
整个系统的核心流程如下：

`流量输入 -> 检测/特征提取 -> 结构化证据 -> RAG + LLM 决策 -> nftables 执行 -> 评估 -> 前端展示`

系统当前支持三类实验通道，它们的流程略有区别。

#### 1.3.1 离线 PCAP 通道
离线通道主要用于复现实验和结果回溯，流程如下：

1. 用户提供 PCAP 文件。
2. 系统调用 Suricata 离线检测该 PCAP。
3. 生成 `eve.json`。
4. 从 `eve.json` 中抽取 `alert` 事件。
5. 对告警做噪声过滤。
6. 按 `src_ip + 时间窗口` 聚合成窗口级证据。
7. 按 `hits`、`severity` 等条件选出 top-k 窗口。
8. 把窗口证据构造成 message，送入 RAG + LLM。
9. LLM 输出结构化策略。
10. 执行器调用 `nftables` 执行动作。
11. 系统保存 job 结果并生成评估报告。

#### 1.3.2 在线回放 / 流式处理通道
在线通道主要用于联机演示和回放验证，流程如下：

1. Suricata 在线监听指定接口。
2. `tcpreplay` 通过 `veth + netns` 回放 PCAP。
3. Suricata 持续生成新的 `eve.json` 告警。
4. 系统维护 `eve.json` 读取偏移量，只读取新增告警。
5. 每隔一个 `window_sec` 周期做一次增量处理。
6. 将当前周期内的告警聚合为窗口证据。
7. 将选中的窗口送入 LLM 做实时策略决策。
8. 立即执行 `nftables` 动作。
9. 回放结束后做收尾补读，生成完整摘要与评估结果。

与普通“回放结束后统一分析”的方式不同，这里实现的是滚动窗口式流式处理。

#### 1.3.3 CSV 结构化流级通道
CSV 通道主要用于快速批量实验和理想输入条件下的策略验证，流程如下：

1. 读取 `TrafficLabelling` CSV。
2. 将每条结构化流量记录映射为统一 window。
3. 自动派生标签、攻击族、严重度等信息。
4. 构造成 message。
5. 注入 RAG 和上下文信息。
6. 进入 LLM 决策链。
7. 触发执行和评估流程。

这个通道的作用更偏向于验证“在理想结构化输入条件下，LLM 能做出什么策略决策”。

### 1.4 核心设计原则
这个系统的整体思路可以概括为以下几个原则：

1. **先结构化，再决策**
   - 不让 LLM 直接处理原始报文，而是先转成结构化证据。

2. **LLM 负责策略生成，不负责无约束执行**
   - LLM 负责输出结构化策略，但执行必须受系统规则约束。

3. **系统必须形成闭环**
   - 不只是做检测或分类，而是包含执行、反馈、评估和展示。

4. **支持多输入通道**
   - 离线、在线、CSV 三条链路共享统一的决策框架。

5. **强调可解释和可评估**
   - 所有关键步骤都产出结构化文件，方便评估和论文分析。

## 2. 系统后端实现方式与功能

### 2.1 后端总体结构
后端主要由以下几个部分组成：

1. `pipeline/`
   - 负责输入处理、告警抽取、时间窗聚合和通道调度。

2. `agent/`
   - 负责 message 构造、RAG 检索、LLM 调用、策略约束和执行器交互。

3. `eval/`
   - 负责结构化评估。

4. `api/`
   - 负责 FastAPI 接口，对前端提供运行、查看和评估服务。

5. `scripts/`
   - 提供统一运行脚本、环境初始化脚本和 nftables 脚本。

### 2.2 后端主通道实现

#### 2.2.1 通道统一入口
系统通过 `backend/scripts/run_channel.py` 作为统一入口，根据子命令选择三类通道：

1. `offline`
2. `replay`
3. `csv`

实际调度逻辑由 `backend/pipeline/channel_runner.py` 实现，核心函数包括：

1. `run_offline_channel`
2. `run_replay_channel`
3. `run_csv_channel`

这意味着虽然系统支持三种输入模式，但最终都统一进入同一套 job 管理和输出体系。

#### 2.2.2 离线通道实现
离线通道基于 `run_offline_detect` 和 `aggregate_time_windows` 实现，主要功能包括：

1. 调用 Suricata 离线读取 PCAP。
2. 生成 `eve.json`、`alerts_raw.jsonl`、`alerts_filtered.jsonl`。
3. 按时间窗聚合告警。
4. 生成 `llm_inputs_selected.jsonl`。
5. 调用 LLM 批量决策。
6. 输出 `channel_summary.json`。

#### 2.2.3 在线回放通道实现
在线回放通道是后端中最复杂的一条链路，主要功能包括：

1. 启动 Suricata 监听接口。
2. 检查 Suricata readiness，避免未启动完成就开始回放。
3. 调用 `tcpreplay` 在指定接口或 netns 中回放 PCAP。
4. 对 `eve.json` 做增量读取，只读取新增 `alert`。
5. 周期性聚合当前窗口的告警。
6. 将窗口送入 LLM 并实时执行策略。
7. 在回放结束后等待 drain 时间，补读 `eve.json` 尾部数据。
8. 保存 `tcpreplay.stdout.log` 和 `tcpreplay.stderr.log`，用于回放故障排查。

这部分当前已经实现了流式窗口处理和回放收尾逻辑，是整个系统后端里工程难度最高的部分。

#### 2.2.4 CSV 通道实现
CSV 通道主要由 `backend/pipeline/csv_flow_adapter.py` 完成。它的功能包括：

1. 读取 CSV 流记录。
2. 做采样和标签分层选择。
3. 为每条流构造统一的 window 输入。
4. 输出 `llm_inputs_all.jsonl` 和 `llm_inputs_selected.jsonl`。
5. 进入统一的 LLM 决策链。

### 2.3 检测与特征处理实现

#### 2.3.1 告警抽取与过滤
`backend/pipeline/suricata_alerts.py` 实现了以下能力：

1. 从 `eve.json` 中提取 `event_type=alert`。
2. 清洗字段，抽取出统一记录格式。
3. 过滤明显噪声告警。
4. 统计 top signatures 和 top source IP。

当前已内置过滤规则，例如：

1. `invalid checksum`
2. `invalid ack`
3. `unable to match response to request`
4. `gzip decompression failed`

#### 2.3.2 时间窗聚合
`backend/pipeline/window_aggregate.py` 实现了按源 IP 和时间窗聚合告警的功能。

聚合结果中包含的关键特征包括：

1. `hits`
2. `severity_min`
3. `alert_density_per_sec`
4. `burst_duration_sec`
5. `unique_dest_ip_count`
6. `unique_dest_port_count`
7. `signature_diversity`
8. `dominant_proto`
9. `top_signatures`
10. `top_categories`
11. `dest_ports`
12. `top_dest_ips`

然后系统根据这些聚合特征选出高优先级窗口送入 LLM。

### 2.4 LLM 决策实现

#### 2.4.1 统一 message 构造
`backend/agent/message_builder.py` 负责将窗口聚合结果转换为统一 message。一个 message 主要包含：

1. `constraints`
2. `hints`
3. `window`
4. `evidence_window`
5. `retrieved_evidence`

这里的 `hints` 是系统从原始窗口二次推导出的高层特征，例如：

1. `attack_family`
2. `noise_ratio`
3. `has_strong_suspicious`
4. `src_ip_private`
5. `top_signature`
6. `top_dest_port`

#### 2.4.2 RAG 检索
`backend/agent/rag_store.py` 负责历史案例存储和检索。

其主要功能包括：

1. 保存历史 message + decision 形成案例库。
2. 用向量数据库检索相似历史案例。
3. 将历史处置结果作为 `retrieved_evidence` 注入当前 message。

RAG 的作用主要是提高策略一致性和可解释性。

#### 2.4.3 LLM 调用与工具调用
`backend/agent/llm_agent_sf.py` 使用 SiliconFlow 的 OpenAI 兼容接口调用模型，并为模型注册工具 schema。

当前向模型暴露的工具包括：

1. `block_ip`
2. `rate_limit_ip`
3. `watch_ip`
4. `unblock_ip`
5. `list_blocked`

模型输出的不是自然语言，而是结构化 JSON 决策对象，并且在需要时可以发起工具调用。

#### 2.4.4 决策保护机制
`backend/agent/policy.py` 提供了运行时策略约束，主要功能包括：

1. 约束可用动作类型
2. 限制 TTL 范围
3. 定义不同攻击族的策略模板
4. 设置 `never_block_ips`
5. 做高噪声流量保护
6. 为 block 行为推荐 TTL
7. 根据攻击族、风险等级和上下文决定执行模式

也就是说，后端不是把流量直接交给 LLM 随意处理，而是通过一套规则对模型进行保护和纠偏。

### 2.5 执行层实现
`backend/agent/mcp_enforcer_server.py` 实现了与 `nftables` 的执行对接。

它的功能包括：

1. 检查 IP 是否已经在对应 set 中。
2. 将 IP 加入：
   - `blocklist_v4`
   - `ratelimit_v4`
   - `watchlist_v4`
3. 支持 `block`、`rate_limit`、`watch` 等动作。
4. 对执行动作做审计记录。
5. 支持 `DRY_RUN` 模式。

这使得后端真正具备了处置能力，而不是只停留在“决策建议”层。

### 2.6 评估模块实现
`backend/eval/channel_eval.py` 实现了结构化评估逻辑，当前可以评估：

1. 动作分布
2. 执行成功/失败情况
3. 重复执行比例
4. 短路执行比例
5. already_present 情况
6. 唯一阻断/限速/观察 IP 数量
7. CSV 模式下的 Precision / Recall / F1
8. 告警压制效果
9. 风险样本和错误样本

最终评估结果会写入 `evaluation_report.json`，供前端和后续论文分析使用。

### 2.7 API 层实现
`backend/api/main.py` 使用 FastAPI 对外提供统一接口。

当前主要接口包括：

1. `/api/health`
   - 健康检查
2. `/api/config`
   - 返回默认参数和环境配置
3. `/api/jobs`
   - 获取历史 job 列表
4. `/api/jobs/{job_id}`
   - 获取单个 job 的完整快照
5. `/api/nft/status`
   - 获取 nftables 三个集合的实时状态
6. `/api/jobs/{job_id}/evaluate`
   - 重新评估某个 job
7. `/api/runs`
   - 查看当前和历史运行记录
8. `/api/runs/{run_id}`
   - 查看单次运行详情
9. `/api/runs`
   - 启动新任务

因此后端已经不仅是本地脚本集合，而是有完整 API 管理层的实验平台。

## 3. 系统前端实现方式与功能

### 3.1 前端技术实现
前端采用以下技术栈实现：

1. React 18
2. Vite
3. 单页应用结构
4. 原生 CSS 进行样式组织

前端入口文件主要包括：

1. `frontend/src/App.jsx`
2. `frontend/src/main.jsx`
3. `frontend/src/styles.css`

前端直接通过 `fetch` 调用后端 FastAPI 接口，不依赖额外的状态管理框架。

### 3.2 前端页面组织方式
前端整体上是一个实验控制台，采用“主页 + 三个通道页”的组织方式。

#### 3.2.1 主页
主页的作用是展示系统概览并让用户选择实验通道，当前支持：

1. 离线通道
2. 在线通道
3. CSV 通道

#### 3.2.2 三个通道页面
每个通道页面都保持统一结构，包括：

1. 实验控制面板
2. 历史实验记录
3. LLM 决策流展示
4. nft 执行流展示
5. nft 实时状态面板
6. 最终结果摘要
7. 评估结果展示

这种设计的好处是：

1. 三种通道共享一致的交互逻辑
2. 又能保留各自特有的参数配置
3. 适合做演示和实验对比

### 3.3 前端交互逻辑实现
前端核心逻辑集中在 `App.jsx` 中，主要通过 React `useState` 和 `useEffect` 管理状态。

前端维护的关键状态包括：

1. 当前页面
2. 三个通道的表单参数
3. job 列表
4. 当前选中的 job
5. 最新 run 状态
6. nftables 实时状态
7. loading 状态

前端通过轮询实现两个关键实时功能：

1. 轮询 `/api/runs/{run_id}`
   - 用于追踪任务执行状态
2. 轮询 `/api/nft/status`
   - 用于展示 nftables 实时状态

### 3.4 前端已实现的功能

#### 3.4.1 实验启动与参数配置
前端可以直接配置并启动三类实验。

离线通道可配置：

1. PCAP 路径
2. 时间窗
3. 最小 hits
4. top-k
5. RAG top-k

在线通道可配置：

1. PCAP 路径
2. Suricata 接口
3. Replay 接口
4. 命名空间
5. 校验模式
6. 回放速度
7. 回放附加参数
8. 抓取等待时间
9. Suricata 就绪超时
10. 聚合参数

CSV 通道可配置：

1. CSV 路径
2. top-k
3. 采样模式
4. 随机种子
5. 是否排除 benign

#### 3.4.2 历史 job 管理
前端可以获取并展示历史实验记录，包括：

1. job_id
2. 通道类型
3. source_path
4. 是否已评估
5. 决策数量
6. 重复执行比例
7. 短路次数

用户可以在前端直接切换不同历史 job，查看对应的实验结果。

#### 3.4.3 决策流展示
前端提供决策时间线组件，用于展示：

1. 每条决策的动作类型
2. 目标 IP
3. TTL
4. 决策状态
5. 执行模式
6. 决策原因

这部分本质上是把 `llm_decisions` 和 job 时间线做了可视化。

#### 3.4.4 执行流展示
前端还单独展示 nft 执行结果，包括：

1. tool 是否成功
2. 是否跳过执行
3. 是否被已有动作覆盖
4. 是否 already_present
5. 执行模式和 TTL

这使前端不只是展示模型结果，还能展示真正的执行层反馈。

#### 3.4.5 nft 实时状态展示
前端会读取 `/api/nft/status`，展示当前三个 nft set：

1. 阻断集合
2. 限速集合
3. 观察集合

并显示：

1. 当前成员数量
2. 每个 IP
3. TTL
4. 剩余时间

这部分是系统演示里非常重要的“真实执行证据”。

#### 3.4.6 最终结果和评估结果展示
前端还实现了结果汇总面板和评估面板，能够展示：

1. 决策总量
2. 样本规模
3. 重复执行比
4. 短路次数
5. Precision / Recall / F1
6. 唯一阻断 IP 数量
7. 动作分布图
8. 执行模式分布图
9. 决策状态分布图
10. TTL 原因分布图
11. 数据覆盖情况
12. 风险样本
13. 压制效果

从实现角度看，前端已经不是简单的“运行按钮 + 文本输出”，而是一个面向实验展示的可视化控制台。

### 3.5 前端设计特点
前端的实现特点主要有三点：

1. **通道化设计**
   - 三类实验通道统一纳入一个控制台，但每个通道有独立配置面板和展示语义。

2. **实时状态感知**
   - 能实时看到任务执行状态和 nftables 状态。

3. **面向汇报和答辩展示**
   - 页面中显式区分了实验控制、决策流、执行流、评估结果和最终结果，适合直接用于阶段汇报和系统演示。

## 4. 总结
MalBlock 当前已经形成一个完整的恶意流量阻断实验系统。

从系统架构上看，它不是单纯的流量分类模型，而是一个由输入、检测、结构化处理、RAG + LLM 决策、`nftables` 执行、结构化评估和前端展示组成的完整闭环平台。

后端已经完成三类通道、LLM 决策链、执行链、评估链和 API 管理层；前端已经完成实验控制台、任务查看、决策流展示、执行流展示、nft 状态展示和评估结果展示。

因此，这个系统目前已经具备较强的研究实现价值，也具备较好的汇报和答辩展示能力。
