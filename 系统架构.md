# MalBlock 系统架构、Agent 设计与实验设计说明

## 1. 项目定位与研究问题

MalBlock 是一个面向**恶意流量处置闭环**的研究型系统。系统重点不是重新训练一个上游恶意流量检测模型，而是在已有检测证据或结构化流量特征的基础上，研究如下问题：

1. 如何把原始告警或结构化流量记录组织成适合大模型理解的决策输入；
2. 如何构建一个具备工具调用能力的 Agent，使其能够输出结构化阻断决策而不是自然语言建议；
3. 如何将 Agent 的决策映射到真实执行层，在 `nftables` 中完成阻断、限速或观察动作；
4. 如何从决策有效性、执行有效性和干预收益观察三个层面，对系统进行实验验证。

因此，MalBlock 的研究对象不是“检测模型本身”，而是**大模型驱动的阻断决策 Agent 及其执行联动系统**。在整个系统中：

- **Suricata** 负责产生上游安全告警，是证据来源；
- **CSV 通道** 负责提供理想结构化输入，用于更稳定地观察 Agent 的决策行为；
- **MCP + nftables** 负责执行联动；
- **评估模块** 负责把实验输出组织成论文可用的结构化结果。

---

## 2. 系统总体架构

### 2.1 分层结构

从代码实现看，系统可划分为七个层次：

1. **输入与实验通道层**
   - Offline PCAP 通道
   - Replay 在线回放通道
   - CSV 结构化流量通道

2. **检测与原始证据层**
   - Suricata 产生 `eve.json`
   - CSV 适配器直接读取结构化流记录

3. **证据加工层**
   - 告警抽取
   - 噪声过滤
   - 时间窗聚合
   - 窗口排序与采样

4. **Agent 输入组织层**
   - 窗口消息构造
   - hints 派生特征生成
   - RAG 历史案例检索
   - 决策上下文注入

5. **Agent 决策与约束控制层**
   - 规则预检查
   - LLM 推理
   - Tool calling
   - 决策后处理
   - Schema 校验
   - 运行时修复

6. **执行联动层**
   - MCP client / server
   - `nftables` 三类集合写入
   - `block` / `rate_limit` / `watch` 落地

7. **评估与展示层**
   - 结构化评估报告
   - FastAPI 后端管理接口
   - React 前端实验控制台
   - replay 对照评估面板

### 2.2 系统主链路

系统的主链路可以概括为：

`输入数据 -> 检测告警/流特征 -> 窗口证据 -> Agent message -> RAG/上下文注入 -> Agent 决策 -> nft 执行 -> 作业归档 -> 评估 -> 前端展示`

如果细化到实现层，其关键中间产物包括：

- `alerts_raw.jsonl`
- `alerts_filtered.jsonl`
- `llm_inputs_all.jsonl`
- `llm_inputs_selected.jsonl`
- `llm_messages.jsonl`
- `llm_decisions.jsonl`
- `channel_summary.json`
- `evaluation_report.json`

它们共同构成了系统完整的实验轨迹。

## 3. 三个实验通道的作用、输入输出与实现逻辑

## 3.1 通道统一入口

三类通道共享统一 CLI 入口：`backend/scripts/run_channel.py`。

该脚本将用户输入映射到 `backend/pipeline/channel_runner.py` 的三个入口函数：

- `run_offline_channel(...)`
- `run_replay_channel(...)`
- `run_csv_channel(...)`

统一入口的意义在于：

1. 三类通道共享统一作业目录结构；
2. 三类通道共享统一 Agent 决策链；
3. 三类通道共享统一评估输出格式；
4. 前端与 FastAPI 只需围绕一套作业管理机制开发。

也就是说，三条通道并不是三套完全独立的系统，而是**同一系统在不同输入条件下的三种实验模式**。

## 3.2 Offline PCAP 通道

### 3.2.1 通道作用

Offline 通道的主要作用是：

1. 在可重复条件下复现实验；
2. 保留接近真实检测链路的证据生成过程；
3. 验证从 PCAP 到告警、再到 Agent 决策与执行的完整闭环；
4. 为论文提供可追溯的案例分析素材。

论文中，Offline 通道主要用于：

- 系统主链路验证；
- 执行有效性验证；
- 典型案例复盘；
- 干预收益观察的辅助验证。

### 3.2.2 输入

Offline 通道的主要输入为：

- `pcap`：待分析的离线流量文件；
- `window_sec`：时间窗长度；
- `min_hits`：候选窗口最小告警数阈值；
- `topk`：送入 Agent 的候选窗口数；
- `rag_top_k`：每个消息检索的历史案例数。

### 3.2.3 实现流程

Offline 通道由 `run_offline_channel(...)` 驱动，核心流程如下：

1. 调用 `backend/pipeline/offline_detect.py` 中的 `run_offline_detect(...)`；
2. 使用 `sudo suricata -r <pcap> -c <conf> -l <job_dir>` 对 PCAP 做离线检测；
3. 生成 `eve.json`；
4. 从 `eve.json` 中抽取 `alert` 事件，输出：
   - `alerts_raw.jsonl`
   - `alerts_filtered.jsonl`
5. 调用 `aggregate_time_windows(...)` 按时间窗聚合；
6. 生成：
   - `llm_inputs_all.jsonl`
   - `llm_inputs_selected.jsonl`
7. 调用 `_run_messages_and_agent(...)` 完成：
   - 消息构造
   - RAG 注入
   - Agent 批量决策
8. 生成：
   - `llm_messages.jsonl`
   - `llm_decisions.jsonl`
   - `channel_summary.json`
9. 作业成功后，可由 API 自动触发或手动调用评估脚本，在评估阶段生成 `evaluation_report.json`。

### 3.2.4 输出

Offline 通道的关键输出包括：

- `eve.json`：Suricata 原始事实来源；
- `alerts_raw.jsonl`：未过滤的告警摘要；
- `alerts_filtered.jsonl`：去噪后的告警摘要；
- `llm_inputs_all.jsonl`：所有窗口候选；
- `llm_inputs_selected.jsonl`：选中窗口；
- `llm_messages.jsonl`：送入 Agent 的消息；
- `llm_decisions.jsonl`：Agent 的最终决策；
- `channel_summary.json`：通道摘要；
- `evaluation_report.json`：结构化评估报告。

### 3.2.5 算法与选择逻辑

Offline 通道的关键算法不在于检测模型，而在于**窗口构造与窗口筛选**：

- 按 `(src_ip, epoch // window_sec)` 聚合；
- 只保留 `hits >= min_hits` 的窗口进入 selected 集；
- 使用 `score_for_rank(...)` 排序，排序元组为：
  - `hits`（越大越优先）
  - `-severity_min`（severity 越小表示越严重，因此越优先）
  - `-len(dest_ip_counter)`（目标越集中越优先）
  - `-burst_duration`（窗口越紧凑越优先）

因此，Offline 通道本质上是**先做告警浓缩和优先级排序，再把高价值窗口送入 Agent 决策**。

## 3.3 Replay 在线回放通道

### 3.3.1 通道作用

Replay 通道的主要作用是：

1. 验证在线回放条件下系统的联动能力；
2. 验证 Agent 决策与执行层能否在连续输入下稳定工作；
3. 验证 `nftables` 写入与回放链路是否可以形成闭环；
4. 为“执行 / 无执行”对照实验提供基础运行模式。

论文中，Replay 通道主要用于：

- 在线联动能力验证；
- 执行有效性验证；
- 干预收益观察；
- 执行组与 DRY_RUN 对照组的差异比较。

### 3.3.2 输入

Replay 通道的主要输入为：

- `pcap`：待回放的流量文件；
- `suricata_interface`：Suricata 监听接口；
- `replay_interface`：`tcpreplay` 回放接口；
- `replay_netns`：回放所在命名空间；
- `suricata_checksum_mode`：Suricata 校验模式；
- `replay_speed`：回放速度；
- `tcpreplay_extra_args`：附加回放参数；
- `capture_wait_sec`：回放结束后的等待时间；
- `suricata_ready_timeout_sec`：就绪超时；
- `window_sec`、`min_hits`、`topk`、`rag_top_k`。

### 3.3.3 在线实验环境

Replay 通道依赖专门的环境初始化脚本：`backend/scripts/init_replay_env.sh`。

该脚本默认创建：

- 命名空间：`mbreplay`
- 主机侧接口：`veth-mb-host`
- 回放侧接口：`veth-mb-replay`
- 默认 MTU：`32000`
- 在两端关闭 `gro/gso/tso/lro` offload

该环境设计的目的包括：

1. 隔离回放链路，避免污染宿主系统正常流量；
2. 让 Suricata 监听路径与 tcpreplay 发送路径明确可控；
3. 减少 veth 环境中因 MTU 与 offload 造成的回放异常；
4. 使 replay 实验具备可复现的部署方式。

### 3.3.4 实现流程

Replay 通道由 `run_replay_channel(...)` 驱动，主要流程如下：

1. 创建作业目录与输出文件；
2. 启动 MCP enforcer client，并实例化 Agent；
3. 使用 `sudo suricata -i <interface>` 启动在线检测；
4. 轮询 `suricata.log`，直到检测引擎 ready；
5. 使用 `sudo ip netns exec <ns> tcpreplay ...` 回放 PCAP；
6. 周期性调用 `_read_new_alerts_from_eve(...)`：
   - 根据文件偏移量只读取 `eve.json` 新增内容；
   - 仅保留 `event_type=alert`；
   - 执行噪声过滤；
7. 累积到一个窗口周期后，调用 `_flush_replay_batch(...)`：
   - 对 pending alerts 做窗口聚合；
   - 选出 selected windows；
   - 调用 `_stream_messages_and_agent(...)` 逐条送入 Agent；
8. 对每个 selected window：
   - `build_message(window)`
   - `add_meta(...)`
   - `inject_rag(...)`
   - `add_decision_context(...)`
   - `agent.run_one(msg)`
   - `append_rag_case(...)`
   - `update_decision_state(...)`
9. 回放结束后继续等待 drain 条件满足；
10. 补读尾部新增告警并做最终 flush；
11. 生成 replay 摘要与 `channel_summary.json`；
12. 后续再生成 `evaluation_report.json`。

### 3.3.5 输出

Replay 通道的关键输出包括：

- `eve.json`
- `alerts_raw.jsonl`
- `alerts_filtered.jsonl`
- `llm_inputs_all.jsonl`
- `llm_inputs_selected.jsonl`
- `llm_messages.jsonl`
- `llm_decisions.jsonl`
- `summary.json`（回放层摘要）
- `tcpreplay.stdout.log`
- `tcpreplay.stderr.log`
- `channel_summary.json`
- `evaluation_report.json`

### 3.3.6 算法与运行特征

Replay 通道的关键算法特征有三点：

1. **增量读取算法**
   - 维护 `eve.json` 文件偏移量；
   - 每轮只处理新增内容；
   - 避免重复读取历史告警。

2. **滚动窗口聚合**
   - 不是逐包实时推理；
   - 而是按 `window_sec` 周期对新增告警做批量聚合。

3. **流式决策状态累积**
   - 对同一 IP 的历史决策状态持续保留；
   - 允许在后续窗口中进行 TTL 升级、已有策略覆盖判断与递进处罚。

因此，Replay 通道最准确的表述应为：

> 基于回放流量和滚动窗口聚合的在线联动处置通道。

## 3.4 CSV 结构化流量通道

### 3.4.1 通道作用

CSV 通道的主要作用是：

1. 在理想结构化输入条件下观察 Agent 的决策能力；
2. 在不依赖 Suricata 的情况下快速批量跑实验；
3. 为模型对比、RAG 对比、采样策略对比提供高效实验入口；
4. 为论文中的决策有效性定量分析提供最稳定的数据基础。

论文中，CSV 通道主要用于：

- 决策有效性实验；
- 风险识别能力分析；
- 强处置能力分析；
- 不同模型或不同配置的对比实验。

### 3.4.2 输入

CSV 通道的主要输入为：

- `csv`：结构化流量 CSV 文件；
- `topk`：抽样数量；
- `exclude_benign`：是否排除 benign；
- `selection_mode`：采样方式；
- `seed`：随机种子；
- `rag_top_k`：RAG 检索条数。

### 3.4.3 实现流程

CSV 通道由 `run_csv_channel(...)` 驱动，核心流程如下：

1. 创建作业目录；
2. 调用 `build_csv_flow_inputs(...)` 读取 CSV；
3. 对每一行调用 `_build_flow_window(...)`，直接构造统一 window；
4. 生成：
   - `llm_inputs_all.jsonl`
   - `llm_inputs_selected.jsonl`
5. 调用 `_run_messages_and_agent(...)`，进入统一 Agent 决策链；
6. 先生成 `channel_summary.json`；若后续触发评估，再生成 `evaluation_report.json`。

### 3.4.4 输出

CSV 通道的关键输出包括：

- `llm_inputs_all.jsonl`
- `llm_inputs_selected.jsonl`
- `llm_messages.jsonl`
- `llm_decisions.jsonl`
- `channel_summary.json`
- `evaluation_report.json`

与 Offline / Replay 通道不同，CSV 通道不会生成 `eve.json` 和告警抽取产物，因为其证据来源本身就是结构化流记录。

### 3.4.5 流记录到窗口的映射逻辑

`_build_flow_window(...)` 会把每条 CSV 记录转换成统一 window，其中包含：

- 源 / 目的 IP 与端口；
- 时间戳与持续时间；
- 协议；
- 包速与流量速率；
- `CSV_FLOW::<label>` 伪 signature；
- attack_family；
- severity；
- `csv_features` 原始扩展字段。

这使得 CSV 通道尽管不经过 Suricata，也能被后续 message builder、RAG、Agent 和评估模块统一处理。

### 3.4.6 采样算法

CSV 通道支持三种采样模式：

#### （1）priority
实现上对候选样本按如下元组排序：

- `label_is_malicious`
- `-severity_min`
- `-flow_packets_per_s`

再按 `reverse=True` 排序并截取前 `topk`。该模式更强调高风险样本优先进入 Agent。

#### （2）random
对候选样本进行随机抽样；若样本量小于 `topk`，则直接全量打乱输出。

#### （3）stratified_label
先按 label 建桶，再在各标签桶内做分层抽样；如果 benign 与 malicious 同时存在，则优先分配接近均衡的抽样配额，再补足剩余 quota。其目的在于：

- 避免单一标签占据过多样本；
- 提高实验覆盖多攻击类型的能力；
- 让决策有效性分析更稳定。

因此，CSV 通道不仅是“直接读 CSV”，而是带有明确采样控制策略的结构化实验入口。

### 模块作用
负责三类通道的总编排。

- 各通道运行参数

- `channel_summary.json`

### 关键算法/机制
- 批量模式和流式模式并存；
- replay 模式下按周期 flush pending alerts；
- 统一 job artifact 组织。

### 模块作用

- Suricata 配置路径

- `eve.json`

## 4.3 `backend/pipeline/suricata_alerts.py`

### 模块作用
从 Suricata `eve.json` 中抽取和过滤告警。

### 输入
- `eve.json`
- 噪声过滤规则

### 实现逻辑
1. 遍历事件；
2. 保留 `event_type=alert`；
3. 提取统一字段，例如：
   - timestamp
   - src_ip
   - dest_ip
   - dest_port
   - severity
   - signature
   - category
4. 使用关键词规则过滤噪声 signature。

### 输出
- 标准化 alert record 列表 / JSONL

### 算法特点
- 基于规则的轻量级去噪；
- 目标是减少明显噪声，而非替代检测。

---

## 4.4 `backend/pipeline/window_aggregate.py`

### 模块作用
将离散告警聚合为窗口级结构化证据。

### 输入
- `alerts_filtered.jsonl`
- `window_sec`
- `min_hits`
- `topk`

### 实现逻辑
1. 按 `(src_ip, epoch // window_sec)` 建立窗口；
2. 为每个窗口维护统计器：
   - signature counter
   - category counter
   - dest_ip counter
   - dest_port counter
   - proto counter
3. 计算统计特征；
4. 导出 ALL 和 SELECTED 两类窗口文件；
5. 生成摘要文件。

### 输出
- `llm_inputs_all.jsonl`
- `llm_inputs_selected.jsonl`
- `llm_inputs_summary.json`

### 核心算法
#### 聚合算法
把同一 `src_ip` 在同一时间窗中的多条告警汇总为单个窗口对象。

#### 排序算法
对满足 `hits >= min_hits` 的窗口按 `score_for_rank(...)` 排序：

- 告警数越多优先；
- `severity_min` 越小（表示越严重）越优先；
- 目标越集中优先；
- 窗口越紧凑优先。

该算法的目的不是直接判断恶意，而是从大量窗口中挑出更值得 Agent 关注的高价值候选。

---

## 4.5 `backend/pipeline/csv_flow_adapter.py`

### 模块作用
把结构化 CSV 流记录映射成统一 window，并完成采样。

### 输入
- CSV 文件
- `include_benign`
- `topk`
- `selection_mode`
- `seed`

### 实现逻辑
1. 读取 CSV 行；
2. 归一化列名与字段值；
3. 调用 `_build_flow_window(...)` 构造统一 window；
4. 根据 benign / malicious 标记筛选候选；
5. 使用三种采样算法之一生成 selected 集；
6. 输出 all / selected 两类窗口文件。

### 输出
- `llm_inputs_all.jsonl`
- `llm_inputs_selected.jsonl`
- 构建结果统计信息

### 核心算法
- 流记录统一 window 映射算法；
- `priority` / `random` / `stratified_label` 三类采样算法；
- 通过文件 offset 机制从 all 输出中回读 selected 样本，避免重复构造对象。

## 4.8 `backend/agent/rag_store.py`

### 模块作用

### 输入

- 决策结果
- RAG 配置

### 实现逻辑
1. 把 message + decision 封装为 historical case；

3. 若向量检索失败，则从 JSONL 归档中按规则评分回退检索；

#### 回写算法

## 4.10 Agent：`backend/agent/llm_agent_sf.py`

### 模块作用
这是系统中真正承担阻断决策任务的 **Agent**。它不是一个简单的单轮大模型调用器，而是一个具备：

- 结构化输入消费能力；
- 工具调用能力；
- 决策约束能力；
- 后处理与修复能力；
- 安全回退能力；

的决策代理。

### 输入
- 完整 message：`constraints + hints + window + evidence_window + retrieved_evidence + decision_context + meta`

### 输出
- 单个结构化决策 JSON，核心字段包括：
  - `action`
  - `target`
  - `ttl_sec`
  - `confidence`
  - `risk_score`
  - `labels`
  - `reasons`
  - `strategy`
  - `evidence`
  - `tool_result`
  - `decision_state`
  - `ttl_reason`

### Agent 决策流程
Agent 的 `run_one(...)` 流程如下：

1. 先执行 `precheck_action(message)`；
2. 若命中预检查规则，则直接得到预检查决策；
3. 否则，注册工具 schema，并向模型发送：
   - system prompt
   - JSON 序列化后的 user message
4. 若模型发起 tool calls：
   - 解析参数
   - 补齐参数
   - 执行工具或短路执行
   - 将 tool result 再送回模型
5. 要求模型最终输出 JSON-only；
6. 若不是合法 JSON，则要求重试一次；
7. 把结果交给 postprocessor 规范化；
8. 若需要但未执行工具，则执行运行时修复；
9. 最后做 schema 校验；
10. 若任一步骤失败，则进入 fallback 保守策略。

### Agent 的核心特点
1. **不是单次文本问答，而是带工具循环的决策代理**；
2. **不是完全自治，而是规则 + LLM + 后处理共同作用**；
3. **不是只给建议，而是直接和执行层联动**；
4. **不是每次独立判断，而是结合当前 job 内历史上下文决策**。

因此，在论文与文档中，完全可以明确将其定义为：

> 面向恶意流量处置的阻断决策 Agent。

## 4.11 `backend/agent/policy.py`

### 模块作用
负责定义决策约束与策略模板。

### 输入
- 当前 message 的 hints、constraints、decision_context

### 实现逻辑
1. 定义允许动作、TTL 范围、执行模式；
2. 定义不同攻击族的策略模板；
3. 计算 escalation level；
4. 推荐 TTL 和 priority；
5. 计算 follow_up；
6. 执行预检查。

### 输出
- 约束字典
- 预检查决策
- 推荐策略信息

### 核心算法
#### 攻击族策略模板
对 `dos`、`scan`、`brute-force`、`botnet`、`suspicious-dns`、`web-attack` 等分别定义：

- base_action
- base_execution_mode
- base_ttl_sec
- priority
- follow_up

#### escalation_level 算法
综合以下因素计算升级等级：

- 历史窗口出现次数；
- 历史 block / monitor 次数；
- 检索到的历史 block 次数；
- 当前 hits 和 severity 风险强度。

#### precheck_action 算法
在进入 LLM 之前对以下情形直接做保守处理：

- `src_ip` 非法；
- 噪声流量；
- 高噪声但无强可疑证据；
- 保护 IP；
- 不允许阻断的私网源 IP；
- 模板建议过强但证据仍混杂。

## 4.12 `backend/agent/decision_postprocessor.py`

### 模块作用
对 Agent 输出做后处理、修复和执行保障。

### 输入
- 模型输出决策
- tool results
- 当前 message

### 实现逻辑
1. 解析和补全工具参数；
2. 统一 target 为当前 `src_ip`；
3. 构造 canonical evidence；
4. 重算或抬升 TTL；
5. 当存在更强当前策略时短路；
6. 若应执行但未执行则补执行；
7. 标注 `decision_state` 与 `ttl_reason`。

### 输出
- 标准化决策 JSON

### 关键算法
#### 短路执行算法
如果当前 IP 已处于更强或同级执行模式，则不重复落表，而是返回：

- `skipped_execution=true`
- `covered_by_existing_action=true`

#### TTL 归一化算法
综合：

- 推荐 TTL
- 历史最大 TTL
- 历史 block 次数
- 同攻击族出现次数
- 同 label 出现次数

对 block 的 TTL 做提升和封顶控制。

#### 运行时修复算法
如果最终 action 和 strategy 要求执行工具，但模型没有实际调用工具，则 postprocessor 代为执行。

## 5. Agent 设计：阻断决策 Agent 的结构与工作机制

## 5.1 为什么要构建 Agent

MalBlock 的目标不是让大模型输出一段安全分析文本，而是让它承担**阻断决策者**的角色。因此，系统中构建了一个 Agent，而不是简单的 prompt + completion 调用。

构建 Agent 的原因有三点：

1. 决策必须是结构化的，便于执行与评估；
2. 决策必须能调用工具，与真实执行层联动；
3. 决策必须处于规则与安全约束之下，不能完全自由生成。

因此，MalBlock 中的 Agent 是一个**受约束、可调用工具、可回退的处置代理**。

## 5.2 Agent 的输入空间

Agent 的输入不是原始流量，而是一个系统构造好的结构化 message，包含：

- `constraints`：系统允许的动作、TTL 范围、保护 IP、执行模式约束；
- `hints`：攻击族、噪声比、强可疑标记、burst、端口等派生特征；
- `window`：压缩后的窗口视图；
- `evidence_window`：较完整证据；
- `retrieved_evidence`：历史案例；
- `decision_context`：当前 job 内该 IP 的历史处置状态；
- `meta`：job_id 与 window_key。

这意味着 Agent 决策并不是“盲推理”，而是建立在充分结构化上下文之上的。

## 5.3 Agent 的动作空间

Agent 的动作空间被限制为：

- `allow`
- `review`
- `monitor`
- `block`

对应执行模式为：

- `none`
- `watch`
- `rate_limit`
- `drop`

工具空间则包括：

- `block_ip`
- `rate_limit_ip`
- `watch_ip`
- `unblock_ip`
- `list_blocked`

因此，Agent 的动作并不是无限制自然语言空间，而是一个小而明确的控制空间。

## 5.4 Agent 的决策循环

一次完整 Agent 决策包括以下阶段：

1. **预检查阶段**
   - 对明显噪声、保护 IP、无效 IP 等情况提前保守处理；

2. **模型推理阶段**
   - 调用大模型进行判断；

3. **工具调用阶段**
   - 如果模型认为需要执行动作，则调用相应工具；

4. **后处理阶段**
   - 修复目标、证据、TTL 与标签；

5. **运行时修复阶段**
   - 若应执行而未执行，则补执行；

6. **Schema 校验阶段**
   - 不合法即 fallback。

这套流程使 Agent 具备了“决策—执行—修正—校验”的闭环特征。

## 5.5 Agent 的优势与边界

### 优势
1. 决策具备结构化可执行性；
2. 可与真实执行层联动；
3. 支持历史上下文与递进处罚；
4. 支持 RAG 注入历史案例；
5. 输出可直接进入评估模块。

### 边界
1. Agent 不是完全自治系统；
2. 决策受规则模板、预检查和后处理强约束；
3. RAG 当前提供的是历史案例参考，而不是自学习反馈；
4. 其研究重点是阻断决策与执行闭环，而非检测模型创新。

## 8. 实验设计

本节重点说明**前期实验设计的作用、目的与对应通道**。实验结果可以后续单独补充，但实验设计本身应在论文和系统文档中提前明确。

## 8.1 实验设计总原则

MalBlock 的实验设计遵循以下原则：

1. 不把三类通道混为一谈；
2. 不用单一指标评价整个系统；
3. 将“决策”“执行”“收益”三个问题拆开验证；
4. 优先围绕论文主线——阻断决策与执行闭环——组织实验。

因此，实验设计按功能被拆成四类实验。

---

## 8.2 实验一：CSV 通道决策有效性实验

### 目的
验证在理想结构化输入条件下，Agent 是否能根据样本证据做出合理的阻断决策。

### 作用
该实验主要回答：

- Agent 是否具备风险识别能力；
- Agent 是否具备强处置能力；
- 不同模型 / 不同 RAG 配置下，决策行为是否存在差异。

### 实验输入
- `mixed_eval_cleaned.csv` 或其他结构化 CSV
- 可配置 `topk`、`selection_mode`、`seed`
- 可配置是否开启 RAG

### 控制变量
可设计以下对比：

1. 模型对比
2. RAG 开 / 关 对比
3. 采样方式对比
4. 是否排除 benign 对比

### 评价重点
- 风险识别 Precision / Recall / F1
- 强处置 Precision / Recall / F1
- 动作分布
- 执行模式分布
- 错误样本分析

### 论文作用
该实验是论文中“决策有效性”的核心定量实验。

---

## 8.3 实验二：Offline 通道闭环验证实验

### 目的
验证系统在离线 PCAP 条件下能否完成从检测、聚合、Agent 决策到执行联动的完整闭环。

### 作用
该实验主要回答：

- Suricata 告警能否被正确转换为窗口证据；
- Agent 是否能消费这些证据并输出结构化决策；
- 决策是否真正落到执行层；
- 哪些窗口会触发 block / monitor / review 等不同动作。

### 实验输入
- 原始 PCAP
- `window_sec`
- `min_hits`
- `topk`
- `rag_top_k`

### 核心观察项
- selected windows 数量
- 决策时间线
- 执行时间线
- nft set 变化
- tool success / failure
- decision-to-execution consistency

### 论文作用
该实验用于证明系统具备真实工程闭环，而不是只做 CSV 上的静态决策模拟。

---

## 8.4 实验三：Replay 通道在线联动实验

### 目的
验证系统在在线回放条件下能否持续接收新增告警、按周期构造窗口并触发 Agent 决策与执行联动。

### 作用
该实验主要回答：

- 在线回放环境是否稳定；
- Agent 在连续输入场景下是否能持续工作；
- 执行层是否能在联机场景下稳定落表；
- 决策上下文与递进处罚机制是否能生效。

### 实验输入
- PCAP
- replay 环境配置
- `window_sec`
- `min_hits`
- `topk`
- `rag_top_k`

### 核心观察项
- `stream_batches` 批次数量
- 每批次 selected windows 数量
- 决策状态（new / escalated / covered）
- 重复处置比例
- skipped execution 数量
- nft 实时状态变化

### 论文作用
该实验用于证明系统不仅能做离线批处理，还能在联机回放环境下形成滚动窗口式实时处置能力。

---

## 8.5 实验四：Replay 执行组 vs 无执行组对照实验

### 目的
观察真实执行动作是否会带来后续风险下降趋势，并将其与 DRY_RUN 基线进行比较。

### 作用
该实验主要回答：

- 执行阻断后，后续风险是否下降；
- 真正执行与无执行之间是否存在差异；
- 这种差异是否能够从平均压制率、告警量差值等指标中被观察到。

### 实验设计
设置两组 replay 实验：

1. **执行组**
   - `DRY_RUN=0`
   - Agent 决策真实写入 `nftables`

2. **无执行对照组**
   - `DRY_RUN=1`
   - 决策流程保留，但不真正落表

再通过：
- `backend/scripts/evaluate_replay_comparison.py`
- `backend/eval/replay_compare_eval.py`

对两组结果进行比较。

### 核心观察项
- 匹配决策数
- 执行组 / 对照组 match rate
- 平均压制率差值
- 过滤告警差值
- 工具成功数差值
- 有效执行差值
- 动作/模式差异样本

### 论文作用
该实验为“干预收益观察”提供最直接的对照证据，是论文中非常重要的一组实验设计。

---

## 8.6 实验五：RAG 辅助效果对比实验

### 目的
验证历史案例检索是否会影响 Agent 的决策一致性、执行模式选择与递进处罚行为。

### 作用
该实验主要回答：

- RAG 是否改变动作分布；
- RAG 是否改变 execution mode 选择；
- RAG 是否影响重复窗口的处理方式；
- RAG 是否改善决策稳定性或解释性。

### 实验设计
在相同通道、相同样本和相同模型条件下，对比：

1. `rag_top_k = 0` 或不注入 RAG
2. `rag_top_k > 0` 注入历史案例

### 核心观察项
- action distribution
- execution_mode distribution
- repeat enforcement ratio
- covered_by_existing_action ratio
- 典型样本决策差异

### 论文作用
该实验用于说明历史案例检索在本系统中的辅助作用，但不应夸大为完整自学习反馈系统。

## 11. 当前系统特点与边界

### 11.1 当前系统特点
1. 三通道统一架构，便于对比实验；
2. 构建了一个面向阻断决策的 Agent，而不是单纯 prompt 调用；
3. Agent 可以调用真实执行工具并联动 `nftables`；
4. 决策链包含规则、RAG、上下文、后处理与校验；
5. 评估输出结构化，适合论文写作和前端展示；
6. 支持执行 / 无执行对照实验。

### 11.2 当前系统边界
1. 在线通道是滚动窗口式流式处理，不是逐事件强实时系统；
2. RAG 是历史案例检索与回写，不是完整反馈学习闭环；
3. `effect_eval` 属于代理收益指标，不应写成严格因果证明；
4. 上游检测能力依赖 Suricata 或结构化 CSV，不是本论文主要创新对象。

---

## 12. 总结

MalBlock 已经形成一个围绕**阻断决策 Agent**展开的系统化研究原型。系统通过 Offline、Replay 和 CSV 三个通道分别承担：

- 决策有效性验证；
- 闭环执行验证；
- 在线联动验证；
- 干预收益观察；
- 不同模型 / RAG / 执行设置的对比实验。

在实现层面，系统已经具备：

1. 从原始告警或结构化流记录构造窗口证据的能力；
2. 构建 Agent message、注入 RAG 与决策上下文的能力；
3. 由 Agent 生成结构化阻断决策并调用工具的能力；
4. 通过 MCP 与 `nftables` 完成真实执行联动的能力；
5. 输出结构化评估与前端展示结果的能力。

因此，从论文定位上看，MalBlock 的核心价值不在于单独某个模型的分类精度，而在于它实现了一套**以大模型 Agent 为核心、面向恶意流量阻断决策与执行闭环的实验平台**。
