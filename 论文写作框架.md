# MalBlock 论文写作框架

## 1. 论文定位与写作主线

### 1.1 论文题目对应的核心问题
你的题目是“基于大模型的恶意流量阻断系统”，因此论文核心不应写成“如何训练一个恶意流量检测模型”，而应写成：

> 在已有检测证据的前提下，如何利用大模型完成恶意流量的阻断决策、执行联动与干预效果观察。

换句话说，论文研究对象是 **阻断决策系统**，不是“原始流量检测器”。

### 1.2 全文建议围绕的三条主线
建议整篇论文始终围绕以下三条主线展开：

1. **大模型能否根据已有安全告警做出合理阻断决策。**
2. **大模型输出的决策能否映射到真实执行层，形成系统闭环。**
3. **执行阻断后，后续风险是否出现下降趋势。**

### 1.3 写作时要刻意避免的偏题方向
以下内容可以写，但不要写成全文中心：

- 不要把重点放在 Suricata 如何检测恶意流量上。
- 不要把论文写成 CIC-IDS2017 分类识别论文。
- 不要把主要结论写成“模型检测准确率提升了多少”。

更准确的表述应该是：

- Suricata 是上游检测证据提供器；
- LLM 是下游阻断决策器；
- 系统价值体现在“决策—执行—收益观察”闭环。

## 3. 各章节具体写什么

## 3.1 第一章 绪论

### 建议写法
第一章重点回答三个问题：

1. 为什么要研究恶意流量阻断，而不只是检测？
2. 为什么要引入大模型，而不是纯规则系统？
3. 你的系统解决了什么具体问题？

### 可以这样概括你的问题背景
传统入侵检测系统能够输出大量告警，但这些告警到实际处置之间仍存在人工分析、策略判断和执行联动的断层。面对复杂、多样、上下文相关的安全事件，仅依赖静态规则往往难以完成细粒度、可解释、可执行的阻断决策。因此，本文尝试在已有检测结果基础上，引入大语言模型作为决策中枢，实现从告警证据理解到阻断动作执行的自动化闭环。

### 本章可突出内容
- 研究背景：检测系统很多，但“告警到处置”自动化不足。
- 研究意义：降低人工分析成本，提高联动响应能力。
- 研究内容：设计并实现一个基于 LLM 的恶意流量阻断系统。
- 主要工作：系统实现、三类通道实验、统一评估框架、案例复盘。

### 本章建议加入的图表
- **表 1-1 国内外相关研究对比表**：从“研究对象、是否使用 LLM、是否包含执行联动、是否关注干预收益”四个角度对比现有工作与本文工作，突出你的研究对象是阻断决策闭环而不是检测模型。

### 建议的创新点/工作量写法
你可以把“创新点”写得务实一些，不要写得过大：

1. 设计了一个基于大模型的恶意流量阻断闭环系统。
2. 实现了告警窗口组织、策略决策、工具执行和状态回写的一体化流程。
3. 构建了适用于该系统的多通道实验与评估框架。
4. 引入历史案例检索增强机制，提高阻断决策参考能力。

## 3.2 第二章 相关技术与理论基础

这一章不要写成百科全书，而是只写支撑你系统实现的关键技术。建议这章的写法遵循“概念定义 → 本文采用方式 → 为后续系统实现提供什么支撑”的逻辑，而不是单纯堆概念。

### 本章建议进一步细化为以下写法层次
- 先定义“检测”与“阻断”的差异，明确本文研究对象是检测后的处置决策。
- 再介绍 Suricata、LLM、RAG、nftables 这四类支撑技术。
- 每一节最后都要落回一句：该技术在 MalBlock 中承担什么角色。
- 本章重点是“理论基础”和“技术选型依据”，不要提前展开第四章中的实现细节。

### 2.1 入侵检测与恶意流量阻断
说明检测与阻断的区别：
- 检测负责发现可疑行为；
- 阻断负责根据风险程度决定是否执行拦截、限速或观察。

这一小节建议展开成三段：
1. **入侵检测的作用**：发现异常流量、输出签名与告警证据，但不直接决定如何处置。
2. **阻断处置的作用**：结合风险程度、资产敏感性和历史行为选择 `block / monitor / review / allow` 等策略。
3. **本文关注点**：不研究从原始流量到恶意标签的检测模型，而研究从“已有告警”到“处置动作”的决策闭环。

这一节最好自然引出一句话：
> 因此，MalBlock 的关键问题不是“能否发现恶意流量”，而是“在已有检测证据的前提下，能否做出合适且可执行的阻断决策”。

### 2.2 Suricata
这里要强调：
- Suricata 在本系统中承担“告警证据生成器”的角色；
- 它不是本文的研究核心；
- 它为 LLM 决策提供签名、严重度、端口、时间窗口等依据。

建议这节写成“告警证据来源”视角：
- `backend/pipeline/offline_detect.py` 负责离线 PCAP 检测；
- `backend/pipeline/suricata_alerts.py` 负责从 `eve.json` 中提取 alert 摘要，并执行基础降噪；
- `backend/pipeline/window_aggregate.py` 负责把离散告警聚合为时间窗口，形成后续 LLM 的输入对象。

你可以明确说明：本文并未修改 Suricata 的检测引擎，而是把其输出的 `signature`、`category`、`severity`、`src_ip`、`dest_ip`、`dest_port` 等字段转化为后续决策证据。

### 2.3 大语言模型与工具调用
这一节重点服务第四章。可以介绍：
- LLM 具备上下文理解和结构化输出能力；
- 通过 function calling / tools 机制，可以把决策映射到执行工具；
- 本项目中通过 JSON 决策结构和工具调用完成阻断动作落地。

对应实现可参考：
- `backend/agent/llm_agent_sf.py`

### 2.4 RAG 与历史案例增强
说明为什么要引入历史案例：
- 同类攻击在不同时间窗口中可能重复出现；
- 历史阻断经验可作为当前决策的补充依据；
- RAG 可以提高决策的一致性和可解释性。

建议这一节写出两层含义：
1. **为什么需要历史案例**：单次窗口证据有限，而历史上相似攻击的处置方式可以为当前决策提供参考。
2. **本文采用什么机制**：`backend/agent/rag_store.py` 同时维护 archive JSONL 与向量检索；在向量库不可用时，还支持 archive fallback，以保证系统仍可检索历史经验。

这里可以简单点出 `build_rag_case()`、`retrieve_evidence()`、`append_rag_case()` 三个核心动作，为第四章做铺垫。

对应实现可参考：
- `backend/agent/rag_store.py`
- `backend/agent/run_agent_batch.py`

### 2.5 nftables 动态执行
说明阻断不是停留在建议层，而是可以落实到真实规则集合。

建议突出三个点：
- 本文不仅输出策略文本，还通过工具调用触发执行动作；
- 执行动作被落到 nftables 对应集合中，而不是停留在“建议阻断”；
- 系统同时支持 `block_ip`、`rate_limit_ip`、`watch_ip` 三类执行方式，对应强阻断、限速和观察。

## 3.3 第三章 系统需求分析与总体设计

这一章是论文的“系统设计总览”。这一章建议回答四个问题：系统为什么需要这样分层、每层解决什么问题、模块之间如何协同、为什么要支持多通道运行。

建议写法是“需求分析 → 总体架构 → 模块职责 → 通道设计”，避免直接写成函数实现说明书。

### 3.3.1 系统目标
建议写成三点：
- 能接收多种来源的安全证据；
- 能基于大模型生成阻断决策；
- 能驱动执行层并输出可评估结果。

可以进一步分成功能需求和非功能需求：

#### 功能需求
1. 支持 CSV、Offline PCAP、Replay 三类输入模式；
2. 支持告警聚合、消息构建、LLM 决策、工具执行、结果评估的完整流程；
3. 支持历史案例检索增强与决策状态累积；
4. 支持前端界面对实验过程和结果进行统一展示。

#### 非功能需求
1. 决策输出应具备结构化、可校验、可执行特性；
2. 系统应尽量避免对基础设施 IP 做误阻断；
3. 实验结果应可复现、可对比、可追踪；
4. 评估报告应支持论文中的定量分析与案例复盘。

### 3.3.2 系统总体架构
建议你画一张总体架构图，分成六层：

1. 数据输入层（CSV / PCAP / Replay）
2. 检测与聚合层（Suricata / 时间窗聚合）
3. 决策输入构建层（message builder）
4. LLM 决策层（LLMBlockAgent）
5. 执行联动层（MCP + nftables）
6. 评估与展示层（evaluation report + frontend）

### 3.3.3 系统模块划分
这里可以按代码模块写，但建议每个模块都按“输入—职责—输出”三句话来写。

#### （1）前端展示与实验控制模块
- 对应：`frontend/src/App.jsx`
- 输入：用户填写的实验参数与作业查询请求；
- 职责：提供 CSV、Offline、Replay 三类实验页面，展示决策流、执行流、最终评估结果；
- 输出：实验请求参数、结果展示界面。

#### （2）后端接口与任务编排模块
- 对应：`backend/api/main.py`
- 输入：前端传入的通道类型、数据路径、窗口参数、回放参数等；
- 职责：统一封装实验命令、启动作业、读取作业结果、返回评估报告；
- 输出：作业记录、状态信息、评估数据。

#### （3）多通道处理与数据准备模块
- 对应：`backend/pipeline/channel_runner.py`、`backend/pipeline/offline_detect.py`、`backend/pipeline/csv_flow_adapter.py`、`backend/pipeline/window_aggregate.py`
- 输入：CSV 文件、PCAP 文件或 Replay 在线流量；
- 职责：对不同来源的数据做统一转换，生成适合后续决策的窗口级输入；
- 输出：`llm_inputs_all.jsonl`、`llm_inputs_selected.jsonl`、中间摘要文件。

#### （4）LLM 决策模块
- 对应：`backend/agent/run_agent_batch.py`、`backend/agent/llm_agent_sf.py`
- 输入：结构化 message、历史案例、决策上下文；
- 职责：调用大模型生成阻断决策，并与工具执行链路联动；
- 输出：`llm_decisions.jsonl` 与对应的工具执行结果。

#### （5）策略约束与安全控制模块
- 对应：`backend/agent/policy.py`、`backend/agent/decision_schema.py`、`backend/agent/decision_postprocessor.py`
- 输入：模型输出决策、上下文约束、历史状态；
- 职责：约束动作类型、TTL 范围、执行模式和受保护 IP，避免不合理或危险输出；
- 输出：规范化、可执行、可校验的最终决策。

#### （6）RAG 历史经验模块
- 对应：`backend/agent/rag_store.py`
- 输入：当前窗口特征、历史案例 archive、向量检索结果；
- 职责：检索相似历史处置经验，并在当前决策完成后将新案例写回历史库；
- 输出：`retrieved_evidence` 与历史案例 archive。

#### （7）执行联动模块
- 对应：`backend/agent/mcp_enforcer_server.py`
- 输入：阻断决策中的 action、execution_mode、ttl、目标 IP；
- 职责：把决策映射为 `block_ip`、`rate_limit_ip`、`watch_ip` 等真实执行动作；
- 输出：工具执行结果与审计记录。

#### （8）统一评估模块
- 对应：`backend/eval/channel_eval.py`、`backend/eval/replay_compare_eval.py`
- 输入：作业目录中的消息、决策、执行结果与中间文件；
- 职责：生成决策有效性、执行有效性、收益观察等统一评估结果；
- 输出：`evaluation_report.json` 与 replay compare 报告。

### 3.3.4 多通道实验模式设计
这是本项目一个比较好写的亮点。

建议先放一张对比表：
- **表 3-1 三类实验通道对比表**：列出“输入对象、是否包含检测环节、是否执行阻断、主要论文用途、主要输出结果”，对比 CSV、Offline、Replay 三类模式。

建议分别写：

#### CSV 通道
- 输入是结构化流量记录；
- 由 `backend/pipeline/csv_flow_adapter.py` 直接将每条流记录转换成窗口级输入；
- 不依赖 Suricata 在线检测，适合快速跑批与标签对齐；
- 重点用于决策有效性验证。

#### Offline PCAP 通道
- 输入是离线 PCAP；
- 由 `backend/pipeline/offline_detect.py` 调用 Suricata 生成 `eve.json`，再由 `suricata_alerts.py` 提取告警、由 `window_aggregate.py` 做时间窗聚合；
- 整体流程是“检测 → 聚合 → 决策 → 执行 → 评估”；
- 适合验证完整闭环。

#### Replay 在线通道
- 输入是在线回放流量；
- 在 `backend/pipeline/channel_runner.py` 中边读取新增告警、边聚合窗口、边触发决策与执行；
- 强调实时性、批次刷新与执行联动；
- 适合展示联机运行效果。

## 3.4 第四章 基于大模型的恶意流量阻断决策实现

这一章最重要，是论文的核心实现章节。建议按“输入如何构造 → 模型如何决策 → 决策如何被约束 → 历史经验如何参与 → 执行如何落地”来展开，形成一条完整实现链路。

### 4.1 告警窗口构建与消息组织
写清楚系统不是直接把原始包送给模型，而是先把告警证据聚合成窗口，再构造成统一消息。
