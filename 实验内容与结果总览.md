- 构建统计：`/home/os/FinalCode/malblock/backend/datasets/cic_ids2017_trafficlabelling/manifest.json`

#### 1.1.2 数据集构建方法
该数据集的构建流程可以概括为以下几个步骤：

1. **遍历 CIC-IDS-2017 TrafficLabelling 全部 CSV 文件。** 构建器会自动读取 `TrafficLabelling` 目录下各工作日对应的原始标注 CSV，而不是只使用某一天的数据。
2. **统一字段命名并保留原始流特征。** 脚本会将原始列名规范化为统一字段，例如 `Source IP`/`Src IP` 统一映射为 `src_ip`，同时尽量保留原始统计特征列。
3. **补充来源与标签元数据。** 在清洗后的结果中额外加入 `source_file`、`source_day`、`original_row_id`、`label_raw`、`label`、`attack_family`、`timestamp_epoch`、`timestamp_minute` 等字段，便于后续实验追踪与分组统计。
4. **执行有效性清洗。** 对缺失 `timestamp`、`src_ip`、`dest_ip`、`label` 的记录直接丢弃；对于时间戳无法解析为标准 UTC 时间的记录也予以剔除。本次构建中共去除了 `288602` 条无效记录。
5. **执行去重。** 当前构建使用 `flow` 去重模式，核心键为：`source_day + label + src_ip + src_port + dest_ip + dst_port + protocol + timestamp_minute`，用于压缩 CIC-IDS-2017 中大量重复或爆发式相邻流记录。
6. **生成恶意主数据集。** 所有非 benign 流量在清洗和去重后汇总为 `malicious_merged_cleaned.csv`，得到 `433935` 条恶意记录。
7. **构造混合评估集。** 再从清洗去重后的良性候选流量中，按 `mixed_eval_benign_ratio = 1.0` 和随机种子 `42` 进行采样，抽取与恶意样本数量相同的良性记录 `433935` 条，与恶意样本合并后形成 `mixed_eval_cleaned.csv`，最终总规模为 `867870` 条。

#### 1.1.3 当前 CSV 数据集的组成特点
根据 `manifest.json`，当前构建结果具有以下特征：
- 原始读取总行数：`3119345`
- 清洗去重后恶意样本数：`433935`
- 混合评估集中的良性样本数：`433935`
- 混合评估集总样本数：`867870`
- 去重模式：`flow`
- 良性采样比例：`1:1`
- 随机种子：`42`

从攻击家族分布看，恶意样本主要由 `dos`、`scan`、`brute-force`、`botnet`、`web-attack` 等类型构成，其中 DoS 与 PortScan 占比最高。因此，这个数据集更适合作为 **CSV 通道下的结构化决策能力评估集**，而不是用来证明模型在原始流量检测上的创新。

### 1.3 Replay 通道实验

当前 Replay 实验共同参数：
- 时间窗口：60 秒
- 最小命中数：3
- 聚合 topk：100
- 回放接口：`veth-mb-replay`

- 回放命名空间：`mbreplay`
- 已启用新的 v5 收益评估逻辑：以执行组中真正生效的 `drop/rate_limit` IP 为目标集，基于两侧 `llm_inputs_all.jsonl` 的 `hits` 字段统计这些 IP 在整个实验中的总告警次数，并按 `R_i=(C_i-E_i)/C_i` 与 `R_total=(ΣC_i-ΣE_i)/ΣC_i` 计算压制率

### 3.2 结果归纳

#### （1）GPT-5.2 无 RAG：整体可用，但误报偏高
`CSV-GPT52` 的风险识别召回率达到 0.986，说明其对恶意流量总体较敏感；强处置 F1 为 0.8166，属于较好水平。但其风险识别精度仅为 0.7436，说明将部分良性流量也纳入了高风险动作范围，整体偏保守。

#### （2）GPT-5.2 + RAG：风险识别提升，但强处置明显变弱
`CSV-GPT52-RAG` 的风险识别 F1 提升到 0.8967，但强处置 F1 下降到 0.6125，强处置召回率仅 0.4839。说明该组实验中，RAG 并未增强强阻断判断，反而使模型变得更谨慎，更倾向于 `review` 和 `allow`，不适合作为论文中的最佳结果。

#### （3）GPT-5.4 无 RAG：整体较均衡
`CSV-GPT54` 的风险识别 F1 为 0.8842，强处置 F1 为 0.8052，且强处置精度高达 0.9912，说明其一旦做出 `block` 决策，错误率很低。该实验的特点是 **阻断动作较稳健、误封较少、整体较均衡**，适合作为 GPT-5.4 的基础对照结果。

#### （4）GPT-5.4 + RAG：风险识别显著增强，但强处置提升不明显
`CSV-GPT54-RAG` 的风险识别 F1 提升到 0.9633，Precision 达到 0.9326，说明 RAG 对风险识别层面有明显帮助；但强处置 F1 为 0.7938，略低于 `CSV-GPT54` 的 0.8052，说明它更擅长“识别出需要关注的风险窗口”，但不一定更积极地给出 `block`。

因此，如果论文强调 **整体风险识别能力提升**，`CSV-GPT54-RAG` 是较好的证据；如果强调 **强处置动作的稳定性**，则 `CSV-GPT54` 更具有代表性。

#### （5）Qwen3-14B 无 RAG：强处置能力极强，但偏激进
`CSV-Qwen3-14B` 的强处置 Recall 高达 0.998，强处置 F1 达到 0.9318，说明其对恶意流量极容易给出 `block`。但其 `block` 动作达到 571 条，明显多于 GPT 系列，风险识别 Precision 也只有 0.7728，说明其整体策略更激进，更接近“宁可多拦也不漏拦”。

这个结果在论文中可以作为一个很有价值的对照：**高强度拦截策略确实能显著提高强处置召回率，但也可能带来更高的过度干预风险。**

#### （6）Qwen14 + RAG：CSV 指标最好，但需谨慎解释
`CSV-Qwen14-RAG` 在当前结果中表现最强：
- 风险识别 F1：0.9692
- 强处置 F1：0.9836
- 强处置 Recall：0.9857
- 强处置 Precision：0.9816

从纯 CSV 决策指标看，它是当前最佳结果。但论文中需要谨慎解释：这说明在当前抽样数据和评估口径下，Qwen14-RAG 的决策输出与标签对齐程度最高；但这并不等价于其在线执行收益一定最优，因为 CSV 通道本身并不评估真实干预收益。

### 3.3 CSV 实验结论

CSV 实验主要可得出以下结论：
1. 所有模型在风险识别层面都具备较高召回率，说明系统已经能够较稳定地区分高风险窗口与普通流量窗口。
2. 不同模型的主要差异体现在 **动作强度分配** 上：有的更谨慎，有的更激进。
3. GPT-5.4 更适合作为论文中的“均衡型 Agent”代表；Qwen 系列更适合作为“强拦截倾向”代表。
4. RAG 对不同模型的影响并不一致，不能笼统地说“加入 RAG 一定更好”。
5. CSV 通道结果最适合支撑论文中的 **决策有效性评估** 一节。

### 4.2 结果归纳

#### （1）Offline 三组实验均完成了“决策—执行—反馈”闭环
三组实验都输出了完整的 `evaluation_report.json`，且 `effect_eval.applicable = true`，说明这些实验不仅完成了决策生成，还进入了执行有效性与收益观察阶段。这使它们能够直接支撑论文中的 **执行有效性评估** 和 **干预收益观察** 两个部分。

#### （2）有效执行数明显低于工具成功数，说明系统具有较强的状态感知能力
例如：
- Tuesday：227 次工具成功，但只有 51 次是新的有效执行
- Wednesday：246 次工具成功，但只有 51 次是新的有效执行
- Thursday：273 次工具成功，但只有 73 次是新的有效执行

这说明系统不是简单地“每条 block 都重复下发一次规则”，而是在相当多场景下识别出了“该对象已经被覆盖”，因此将其归类为重复执行或已有动作覆盖。这一现象对论文是正面的，因为它体现了 Agent 与执行层之间存在 **状态约束与短路机制**，不是无脑重复联动。

#### （3）Thursday 结果最好，适合作为 Offline 主展示实验
`Off-GPT54-Thursday` 的特点最突出：
- 决策到执行一致性最高：0.7326
- 有效执行数最高：73
- block 数最多：219
- mean suppression ratio：0.0563

虽然 Wednesday 的平均压制率略高（0.0659），但 Thursday 在执行数量、执行一致性和整体链路完整度上更均衡，更适合在论文正文中作为 Offline 主案例展示。

#### （4）Offline 收益指标整体存在，但幅度不大
三组 Offline 实验的平均压制率约为：
- Tuesday：5.44%
- Wednesday：6.59%
- Thursday：5.63%

这表明执行动作之后，后续窗口中的告警确实出现了一定下降，但下降幅度并不是特别剧烈。这种结果反而更真实，说明系统已经观察到 **可量化但有限的抑制收益**，而不是夸大的“完全清除”。论文中建议把这一点写成“存在可观测收益，但收益幅度受攻击持续性、窗口划分方式和流量结构影响”。

### 4.3 Offline 实验结论

Offline 三组实验已经足以支撑论文中的以下论点：
1. Agent 输出的结构化动作可以稳定进入执行层；
2. 执行层能够避免大量重复联动；
3. 系统能够在执行后观察到一定程度的后续告警压制；
4. `Off-GPT54-Thursday` 最适合作为离线执行闭环展示的主结果。

### 5.1 关键指标总表

| 实验 | 角色 | 决策数 | 动作分布（block / monitor / review / allow） | tool_success_count | effective_enforcement_count | repeat_enforcement_count | 一致性 | effect_eval 口径 | 备注 |

| Replay-GPT54-Wednesday | 执行组 | 415 | 211 / 51 / 151 / 2 | 262 | 57 | 205 | 0.7099 | v5：执行组目标 IP 告警次数观测 | 单作业仅观测，`evaluated_ips = 17` |
| Replay-GPT54-Wednesday-NO | 对照组 | 417 | 267 / 52 / 97 / 1 | 319 | 130 | 189 | 0.6646 | v5：对照组单作业不定义目标集 | 单作业 `evaluated_ips = 0` |

补充说明：在新的 v5 口径下，单作业 `effect_eval` 不再表示“处置前后 TTL 内变化”，也不再直接给出最终压制率。执行组单作业报告只负责记录**目标 IP 在本组 `llm_inputs_all.jsonl` 中的总告警次数观测值**；正式压制率仅在执行组与对照组的 `suppression_comparison` 中计算。

### 5.2 结果归纳

#### （1）Replay 实验已经完成按告警次数的新口径重评估
当前两个 Replay 作业都已重新生成 `evaluation_report.json`，执行/无执行对照报告也已重新生成。新的收益评估统一切换到 `report_version = "v5"`，其核心口径为：**以执行组中真正生效的 `drop/rate_limit` IP 为目标集，在执行组和对照组各自的 `llm_inputs_all.jsonl` 中读取这些 IP 的 `hits` 总和，并按单 IP 计算压制率，再汇总整体压制率。**

对应公式为：
- 单 IP 压制率：`R_i = (C_i - E_i) / C_i`
- 总体压制率：`R_total = (ΣC_i - ΣE_i) / ΣC_i`

其中，`E_i` 表示执行组中目标 IP 的总告警次数，`C_i` 表示对照组中同一目标 IP 的总告警次数。

#### （2）本次 Replay 对照结果显示压制率为 0
本次 Wednesday Replay 执行/无执行对照重算后的关键结果如下：
- 目标 IP 数：`17`
- 可比较 IP 数：`17`
- 执行组平均告警次数：`290.0`
- 对照组平均告警次数：`290.0`
- 平均告警次数差值：`0.0`
- 执行组总告警次数：`4930`
- 对照组总告警次数：`4930`
- 总告警次数差值：`0`
- 单 IP 平均压制率：`0.0`
- 总体压制率：`0.0`

这说明在当前这组在线回放实验中，**执行组与对照组在目标 IP 告警次数上没有观测到差异**。因此，按照新的正式口径，本次 Replay 实验未体现出可量化的压制收益。

#### （3）单 IP 层面也没有出现改善样本
在本次 `suppression_comparison` 中：
- `improved_pairs = 0`
- `unchanged_pairs = 17`
- `worse_pairs = 0`

这表明 17 个可比较目标 IP 在执行组与对照组中的总告警次数完全一致。也就是说，当前结果不是“有些 IP 变好、有些 IP 变差”，而是**整体没有观察到告警次数层面的变化**。

#### （4）执行组目标集定义与对照组单作业评估的差异仍需单独说明
在新口径下，目标 IP 集合仍然由执行组中真正生效的 `drop/rate_limit` 决策定义，因此：
- 执行组 `effect_eval.evaluated_ips = 17`
- 对照组 `effect_eval.evaluated_ips = 0`

这并不表示对照组没有数据，而是因为**对照组单作业自身不负责定义目标集**。真正有意义的收益比较发生在 `evaluate_replay_comparison.py` 输出的 `suppression_comparison` 中，即以执行组目标集为统一标准，对两侧的总告警次数进行对照统计。

### 5.3 Replay 实验结论

Replay 实验目前可支持以下更稳妥的表述：
1. 系统已经具备在线回放、真实执行和执行/无执行对照评估能力；
2. 新的 v5 收益评估口径不再依赖窗口次数，而是直接比较目标 IP 在整个实验中的总告警次数；
3. 当前 Wednesday Replay 对照结果显示执行组与对照组的目标 IP 告警次数完全一致，因此本次实验未观察到压制收益；
4. 因此，Replay 更适合作为在线补充验证与局限性讨论，而不应单独承担论文中的收益主结论。

### 6.3 当前最值得强调的论文结论

根据现有全部实验，可以较稳妥地提炼出以下结论：

1. **MalBlock 已经具备稳定的决策生成能力。** 在 CSV 通道中，不同模型都能够较高召回地识别风险窗口。
2. **不同模型体现出不同的阻断风格。** GPT-5.4 更均衡，Qwen3-14B 更激进，Qwen14-RAG 在当前 CSV 指标下最优。
3. **系统已形成完整的执行闭环。** Offline 与 Replay 实验均表明，模型输出可以被转换为真实联动动作。
4. **状态感知机制有效减少了重复下发。** 多数组工具调用属于已有规则覆盖，而不是重复创建规则。
5. **执行后可观察到一定程度的告警压制。** Offline 三组实验的平均压制率约在 5%--7% 之间，说明系统存在可量化但有限的实际收益。
6. **在线回放收益评估仍受环境噪声影响。** Replay 对照实验链路已打通，但尚不足以单独支撑最强论文结论。
